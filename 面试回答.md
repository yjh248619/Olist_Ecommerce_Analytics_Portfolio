Q: 为什么 order_items 用复合主键 (order_id, order_item_id)?
A: 业务上"查某订单的所有商品"远多于"查序号为 Y 的所有商品",
   把 order_id 放前面让前缀查询走主键(InnoDB 最左前缀原则)。
   另一个原因:CSV 中确实存在 (order_id, order_item_id) 组合唯一性,
   不能单用 order_id(一对多)。
Q:为什么经纬度用 DECIMAL(10,7) 而不是 FLOAT
A:FLOAT 是浮点数有精度损失：地理坐标的小误差会导致定位偏移几十米
DECIMAL(10,7) = 整数 3 位 + 小数 7 位，精度 = 1.11 cm（1° 纬度 ≈ 111 km，10^-7 度 ≈ 1.11 cm）
10 位总数容纳 ±999.9999999，覆盖全球经纬度（-180180、-9090）
Q:为什么建 (state, city) 复合索引?
A:业务上经常 WHERE state='SP' AND city='são paulo' 查特定城市
复合索引 (state, city) 同时支持 WHERE state=X 单条件 和 WHERE state=X AND city=Y 双条件（最左前缀原则）
不需要单独建 state 索引（被复合索引前缀覆盖）
面试常考：「最左前缀原则」标准考点
Q:为什么要加自增 id 做物理主键
A:InnoDB 表必须有聚簇索引：不显式建主键，会用第一个 UNIQUE NOT NULL 字段；都没有就自动生成 6 字节隐藏 row_id
隐藏 row_id 的问题：
你看不见，无法在 SQL 里引用
全库共享一个递增计数器，高并发插入会有性能瓶颈
一旦未来要删除某条特定记录，没有 id 你定位不到
显式加 id BIGINT UNSIGNED AUTO_INCREMENT 解决全部问题

面试问答
Q: LOAD DATA INFILE 遇到 CSV 里有空字符串塞数字列怎么办?

A: MySQL 默认不会把 '' 当 NULL,会直接报 Incorrect integer value 拒绝。
   解决:用 @变量 暂存 + SET NULLIF(@var, '') 规范化。
   
   背后原理:LOAD DATA 字段列表里,列名 = 直接入表,
   @变量 = 进暂存区不入表,然后 SET 子句做 SQL 表达式转换后再入表。
   
   这是 ETL 的 "T" (Transform) 步骤的最简实现。
LOAD DATA INFILE 'C:/path/to/mysql_uploads/olist_products_dataset.csv'
INTO TABLE products
CHARACTER SET utf8mb4
FIELDS TERMINATED BY ',' OPTIONALLY ENCLOSED BY '"'
LINES TERMINATED BY '\n'
IGNORE 1 LINES
(product_id,
 @category,
 @name_len,
 @desc_len,
 @photos,
 @weight,
 @length,
 @height,
 @width)
SET product_category_name      = NULLIF(@category, ''),
    product_name_lenght        = NULLIF(@name_len, ''),
    product_description_lenght = NULLIF(@desc_len, ''),
    product_photos_qty         = NULLIF(@photos,   ''),
    product_weight_g           = NULLIF(@weight,   ''),
    product_length_cm          = NULLIF(@length,   ''),
    product_height_cm          = NULLIF(@height,   ''),
    product_width_cm           = NULLIF(@width,    '');
逐行讲 why
1. 为什么用 @变量
@category 这种语法叫 MySQL 用户变量(user variable)。LOAD DATA 的字段列表里,如果直接写列名,数据会立刻塞进列;写 @xxx,数据先暂存到变量里,不直接入表,然后用 SET 子句做加工后再入表。

💡 为什么需要这层中转:因为我们要在数据进入数字列之前判断"这是不是空字符串",空字符串要转成 NULL。如果直接写 product_name_lenght,数据已经在尝试入列时就崩了,没机会判断。

2. 为什么用 NULLIF(@xxx, '')
NULLIF(a, b) 的语义:如果 a == b,返回 NULL,否则返回 a。

NULLIF(@name_len, '') 就是:如果 @name_len 是空字符串,变成 NULL;否则原样保留。

NULL 是合法的 SMALLINT UNSIGNED 值(因为你 DDL 允许 NULL),空字符串 '' 才是非法的。

💡 NULLIF 是面试常考点:

它是 CASE WHEN a = b THEN NULL ELSE a END 的语法糖
配合 COALESCE 是处理脏数据的标准组合拳
大厂数据清洗 SQL 里到处都是 NULLIF + COALESCE
3. 为什么 8 个数字字段全部包一层 NULLIF,而不是只包出问题的那个
虽然现在只报了 product_name_lenght 的错,但 LOAD DATA 只报第一个错就停。如果你只修这一个字段,跑下一次它会报 product_description_lenght 同样的错,然后是 product_photos_qty... 要修就一次性全修,省得你来回跑 5 次。

💡 工程思维:报错只是冰山一角,要识别报错背后的规律(这批字段都是基于 category 计算的可空数值),一次性覆盖所有同类风险。这就是工业界"防御性数据导入"的标准思路。

4. category_name 为什么也要包
product_category_name DDL 是 VARCHAR(50) NULL,允许空。理论上字符串列不会因为空字符串报错('' 是合法 VARCHAR 值)。

但是,我建议你保留 NULLIF(@category, ''),把空字符串规范化为 NULL。原因:

后续做分析 WHERE product_category_name IS NULL 比 WHERE product_category_name = '' OR IS NULL 简单
业务语义上,"没有类别"应该是 NULL,不是空字符串(这两个在语义上不同)
工业界数据规范化原则:字符串空值统一 NULL,避免 '' 和 NULL 共存
Q: products 表里 610 个商品没有 category_name,你怎么处理?

A: 不删除,不用 "unknown" 填充,而是规范化为 NULL。
   原因:
   1. 删除丢失业务信息(这些商品有真实订单,删了影响销售分析)
   2. 填充 "unknown" 会污染统计(类别分布里多个虚假大类)
   3. NULL 保留"缺失"语义,后续分析时:
      - 描述性统计:WHERE category IS NOT NULL 排除
      - 冷启动场景:WHERE category IS NULL 单独研究
      - 推荐系统:把"无类别"作为一种特征(可能反映商家不规范上架,
        是质量信号)

Q: 你这个项目里用了 LOAD DATA INFILE,为什么 order_reviews 改用了 Python?

A: LOAD DATA INFILE 性能极高,但对脏文本(字段内换行、引号转义)处理能力有限。
   order_reviews 里用户评论包含未规范化的换行符,LOAD DATA 会字段错位。
   
   决策思路:
   - 其他 8 张表数据规范,坚持 LOAD DATA(单表导入 < 30 秒)
   - order_reviews 单独走 pandas + to_sql(分批 INSERT,~2 分钟)
   - 这是工业界 ETL 的典型权衡:速度 vs 鲁棒性,根据数据特征选工具
   
   面试官追问:为什么不全部用 Python?
   答:LOAD DATA 比 INSERT 快 10-100 倍,大批量规整数据优先用。
       工业界生产 pipeline 也是这样:Sqoop / Flink CDC 跑规整数据,
       脏数据走 Spark 清洗后再 batch insert。
            






背景:order_reviews 表 99224 行,用 LOAD DATA INFILE 导入时报错
      "Incorrect datetime value: 'f63f9a7699...' for column 'review_creation_date'"

诊断:报错值是 32 位十六进制 ID,本该是 DATETIME。说明字段错位。
      根因:用户评论 review_comment_message 字段内含未规范化换行符 \n,
      LOAD DATA 把字段内的 \n 当成行结束符,后续字段连锁错位。
      OPTIONALLY ENCLOSED BY '"' 救不了,因为部分行字段无引号。

决策:其他 8 张表(规整数据)继续 LOAD DATA(快、单表 < 30 秒)
      order_reviews 改用 pandas.read_csv + DataFrame.to_sql(慢但鲁棒)
      
      理由:LOAD DATA 是字符扫描器,pandas 是带状态机的 CSV 解析器(RFC 4180)
            工业界 ETL 也是这样分工:Sqoop / LOAD DATA 跑规整数据,
            脏数据走 Spark/Pandas 清洗后再批量 INSERT

实施:pip install pandas sqlalchemy pymysql
      to_sql(chunksize=5000, method='multi')分批 batch INSERT
      关键:replace('', pd.NA) 把空字符串规范化为 NULL
            pd.to_datetime(errors='coerce') 容错时间解析

验证:CSV 99224 行,MySQL 99224 行,零丢失

反思:如果数据量是 1 亿行而非 10 万,pandas 内存装不下,
      会改用 Spark / Polars 分块流式处理。

Q: 为什么数据导入后第一件事是做数据质量检查,而不是直接开始分析?

A: 在脏数据上跑分析,产出的结论是不可靠的。
   工业界标准流程:
   1. Profile(描述性统计 + 异常检测)
   2. Clean / Mark(清洗 or 标记)
   3. Constrain(上约束防退化)
   4. Analyze(分析)
   
   跳过 1-3 直接做 4,后果是:
   - 报告里的"复购率 30%" 可能是孤儿订单虚高的
   - 推荐系统训练样本里混入异常评分,模型偏差
   - 业务方发现数字不对,你需要全部回滚
   
   数据从业者第一性原则:Garbage In = Garbage Out。

Q: 你在 Olist 项目里发现了什么数据质量问题?

A: 我做了系统化的数据质量检查(7 个外键关系全检),
   发现 products 表中有 13 个 product_category_name 在翻译表中无对应英文。
   
   分析根因:翻译表是人工维护,事实表后续新增类别没同步。
   这是数仓"维度表滞后于事实表"的经典场景。
   
   决策权衡:
   - 方案 A(补翻译):数据保真,适合小批量,我最终选这个
   - 方案 B(置 NULL):丢业务信息,不推荐
   - 方案 C(不加外键):放弃约束,适合大规模外部维度表
   
   我选 A 的原因:13 条可手工补;此项目是分析场景,需要保留所有类别。
   补完后所有外键约束完整加上,事实表与维度表强一致。
   
   收获:理解了"事实表与维度表的同步"是工业界数仓核心问题,
        对应到大数据领域是 SCD(Slowly Changing Dimension)。

Q: 你这个项目里做异常值检查,用什么方法?

A: 我用了分层检查策略:
   1. 硬规则:price <= 0、freight < 0、score 不在 [1,5] —— 这些是"绝对错误"
   2. 业务边界:product_weight_g = 0 (合法数字但业务上=未填,
                应规范化为 NULL)
   3. 统计异常:按 product_category 分组用 IQR 法找离群点
                (不能用全局阈值,因为家电 vs 图书的合理重量范围差 100 倍)
   
   发现 5 条重量异常,通过钻取分析:
   - X 条是 weight=0(数据录入缺失,建议改 NULL)
   - X 条是真实大件家具(category 合理,不是异常)
   
   收获:异常检测**不能拍脑袋定阈值**,要按业务分层。
        工业界做风控/反欺诈也是这个套路:全局规则 + 分层细分。
Q: 你做 Olist 项目时,有没有遇到一个让你印象深刻的 BUG?

A: 第一轮 LOAD DATA 时,我在 SET 子句前不小心多打了一个分号,

   3. 工业界用 Great Expectations、dbt test 这种工具

Q: 8320 条收货前评价的地域分布说明了什么?

A: 两条规律:

   规律 1:所有州无一例外,收货前评价都低于收货后(差 1~2 分)。
         这证实了系统性问题 —— Olist 在发货后就发评价邀请,
         而非签收后。

   规律 2:早评率越高的州,评分越低。
         AL(21%早评) → 2.13分 vs PR(6%早评) → 3.35分。
         早评率高 = 配送时间长 = 用户在等待中收到邀请 = 愤怒评分。

   反直觉发现:RJ(里约)早评 2.24 分,MT(内陆农业州)早评 3.31 分。
   不是"偏远=分低",而是"配送时长超出预期=分低"。
   RJ 用户期待 3 天到,等了 7 天 → 给 2 分。
   MT 用户期待 15 天到,等了 20 天 → 给 3 分。
   期待差而非绝对时长,才是评分的关键。

   对推荐系统的启示:
   如果用评分直接训模型,不是"偏远地区商品被歧视",
   而是"配送时间长的商品被歧视"。
   家具(本身就配送久)和预售商品会被系统性降权,
   不管商品本身多好。

   决策:GMV 分析用 INNER JOIN order_items,
        自动排除无商品行的订单。
        这不是数据损失,而是正确过滤。


        
Q: 介绍一下你做 Olist 项目漏斗分析的核心发现。

A: 三个发现:
   1. 整体转化率 97%,看似高,但绝对值上 2965 单失败,
      损失 GMV ~47 万雷亚尔。最大损耗点不是物流,是付款后缺货(609 单 unavailable)。

   2. 平均配送 11.6 天(备货 2.3 + 物流 8.9)。
      8.9 天的配送时长是 Olist 的"天花板"——

   3. 两年内 shipping_rate 从 87% 提升到 99.6%,

                              PART 5 RFM 用户分层

Q: 你做 RFM 时遇到过什么坑?

A: 第一版我直接套 NTILE(5) 三个维度,跑完发现核心用户 avg_monetary=56,
   流失用户 avg_monetary=310,完全颠倒。
   
   定位到 3 个问题:
   1. R 用 ASC 排序错了 —— recency_days 越小越好,应该是 DESC,
      让最近购买的用户排在后面拿到 NTILE 的高分(5)。
   2. M 用 DESC 错了 —— monetary 越大越好,应该是 ASC,
      让消费最高的用户排在后面拿高分。
   3. F 用 NTILE 完全失效 —— Olist 96.9% 用户 frequency=1,
      NTILE 强行把 93,099 个同频用户切成 5 组,业务含义不稳定。
   
   修正:F 改用业务阈值
       CASE WHEN frequency=1 THEN 1
            WHEN frequency=2 THEN 3
            WHEN frequency BETWEEN 3 AND 5 THEN 4
            ELSE 5
       END AS f_score
   
   收获:NTILE 适合连续分布,业务阈值适合离散+长尾分布。
        套模板之前必须先看数据形状。


Q: 为什么 RFM 必须用 customer_unique_id 而不是 customer_id?

A: Olist 的 customers 表设计很特殊:customer_id 是订单维度的客户 ID,
   每次下单生成一个新的;customer_unique_id 才是真实用户。
   
   全表 99,441 个 customer_id,但只对应 96,096 个真实用户。
   
   如果用 customer_id 做 RFM:
   - 每个 ID 永远只对应 1 个订单
   - frequency 永远=1
   - 复购率永远=0
   - 整个 RFM 模型失去意义
   
   这反映了一个一般原则:做用户分析前必须先确认"用户"这个实体的真实粒度。
   电商系统里"用户"可能是 device_id / login_id / phone_md5 / unique_id,
   不同口径对应不同业务问题。


Q: Olist 复购率 3.1% 这么低,怎么解读?对推荐系统有什么影响?

A: 三层解读:
   
   1. 不是 Olist 不好,是品类决定的天花板:
      - 生鲜电商:复购率 50%+
      - 服饰:20-30%
      - 美妆:15-25%
      - Olist(家具/家电/床品为主):3.1% — 行业正常水平
      家具买完一个沙发不会一年再买第二个。
   
   2. 这反过来定义了 Olist 是"获取型业务",不是"留存型业务":
      - 83.3% 用户是一次性购买
      - 核心用户只有 6 人(F>=3 且 R<=180 天)
      - 39.9% 是新用户,43.4% 是普通用户(买过但没回来)
      增长全部来自拉新,不是留存。
   
   3. 对推荐系统的根本启示:
      - 不优化"老客 LTV",优化"新客首单 CVR"
      - 不做"用户行为序列建模",做"冷启动召回"
      - 不能照搬 MovieLens / 淘宝那套 CF 方案
      - 候选池要基于"品类热度 + 地域偏好 + 卖家健康度"做冷启动
      
      这是和淘宝/京东推荐系统完全不同的设计起点。


Q: 用户分层做出来,核心用户只有 6 个人,这个分层还有意义吗?

A: 有意义,但运营落地方向不一样。
   
   传统电商运营思路:"识别 VIP 给特殊待遇"
   Olist 应用版本:"防止新用户变流失"
   
   分层揭示的真实结构:
   - 新用户 39.9% → 首单转化的关键人群
   - 普通用户 43.4% → 已经买过没回来,最大可挽回群体
   - 流失用户 16.5% → 完全流失,不值得召回
   - 核心 + 中坚 = 0.2% → 太小,不是主战场
   
   所以正确的运营动作:
   - 推荐系统优化新用户首单 CVR
   - 用户成功干预聚焦"首单后 30 天"窗口,防止变成普通用户
   - 不要花 30% 营销预算在 6 个核心用户身上
   
   这是数据分析师的价值 —— 不是套用 RFM 模板,而是用数据
   推翻"先入为主的运营假设"。


Q: 钻取那个买了 15 次的最忠诚用户,你发现什么?

A: 他在 SP 州,总消费 879 雷亚尔,品类横跨工具、运动、时装包。
   平均客单 ~59 雷亚尔,跨度大、品类杂、单价不高。
   
   这不像普通 C 端消费者,更像是:
   - 小 B 商家(在 Olist 批发进货的小店主)
   - 或代购/批发型用户
   
   类似地,Top 6 核心用户里:
   - 多数集中在 SP / RJ / PE 等大城市
   - 品类组合常有"建材 + 装饰 + 家居"或"床品 + 厨房"
   - 客单价 50-150 雷亚尔,与典型 B2B 进货模式吻合
   
   工业界含义:平台需要识别这类"伪 C 用户",
   单独开 B2B 频道或 batch 订单流程,
   既能服务好他们,也避免污染 C 端推荐数据。

Q: cama_mesa_banho 8432 人评分只有 3.91,你怎么解读?

   2. 但平均评分只有 3.91 分,远低于美健类的 4.18
   3. 平均配送 12.76 天,接近全平台均值
   4. 数据质量检查里发现 5 条 weight_g=0 的异常商品,全部是 cama_mesa_banho

   商品维度信息(重量、尺寸)经常缺失,导致:
   - 物流计算运费时只能给"标准重量",实际重量超了拒收
   - 商品详情页用户无法判断尺寸是否合适
   - 收货后实物不符预期,直接打低分
   
   这就是 RFM 看不到的供给侧问题。
   要解决"首单流失",得先治理 cama_mesa_banho 卖家

   面试官追问:你这是相关性还是因果?
   答:相关性。要证明因果需要 AB 实验:
       在 cama_mesa_banho 品类强制商家补全商品维度信息,
       对照组保持现状,7 天后看 review_score 提升幅度。
       这是我后续 AB 实验设计的第 4 个候选实验。

A: 这是 Top 20 品类里配送最慢、评分最低的品类。

   - 配送 20.7 天 vs 全平台均值 11.6 天 → 慢了 78%
   - 评分 3.52 vs 全平台 4.09 → 差了 0.57 分
   - 这两个数字大概率有因果(物流慢导致差评)
   
   决策建议:不是"治理这个品类",而是
   "在推荐系统里降权这个品类的新商品曝光"。
   
   理由:
   - 商品本身可能没问题,是物流性质决定的(大件、定制)
   - 强行优化物流时长经济上不划算
   - 应该让对配送时长不敏感的用户(批发、装修计划性用户)看到
   - 普通用户少看到 → 减少冲动下单后的"等不及"差评
   
   这是把数据洞察转换为推荐策略的标准动作:
   不是治理供给,而是匹配供给与需求。

## Part 7: Cohort 留存 + LTV 分析

### Q1: RFM 已经做了用户分层,为什么还要做 Cohort?

A: RFM 和 Cohort 是从两个正交维度看同一个问题:

- **RFM 是横截面快照**:告诉我用户"现在是什么状态"——新用户 39.9%、普通 43.4%、流失 16.5%
- **Cohort 是纵贯线电影**:告诉我用户"随着时间的推移怎么变化"——次月还有多少人回来?LTV 在涨还是在停滞?

举个例子:RFM 说"新用户占 39.9%",但这 39.9% 是动态的——他们下个月可能变成"普通用户"或"流失用户"。RFM 看不到这个转化过程,Cohort 能看到。

面试官问你"RFM 和 Cohort 有什么区别",核心答案就是:**一个是状态,一个是过程。两者结合才能完整描述用户生命周期。**

### Q2: Olist 的次月留存率只有 0.5%,这正常吗?

A: 正常,而且这本身就是重要结论。

Olist 是获取型业务,不是留存型。证据链:
1. RFM:96.9% 用户只买一次
2. Cohort:次月留存 0.4%-0.7%,17 个月后 LTV 只涨 3.7%
3. 两个独立方法得出同一个结论 → 置信度很高

对比参考:淘宝/京东的次月留存率在 20%-30%,亚马逊 Prime 在 60%+。Olist 0.5% 的留存率不是"数据有问题",而是**商业模式决定的**——巴西电商市场还在早期,平台首要任务是拉新而不是留存。

如果面试官追问"这么低的留存怎么办",回答应该是:低留存本身不是问题,要看获客成本(CAC)和首单利润。如果 CAC < 首单毛利,单次交易就是盈利的,留存可以以后再说。

### Q3: 不同 cohort 的 LTV 差异很大吗?

A: 不大,而且你要区分"真差异"和"窗口效应"。

| 指标 | 范围 | 解读 |
|---|---|---|
| 首单 ARPU | 150-175 BRL | 各 cohort 非常稳定,用户质量一致 |
| 次月留存 | 0.4%-0.7% | 同上 |
| LTV 增幅 | 3-7 BRL | 绝对值差异很小,主要受观测窗口长度影响 |

关键点:**不能用 final_ltv 直接对比不同 cohort,因为观测窗口不一样。** 2017-01 有 19 个月复购累积,2018-08 只有 1 个月。正确的对比方式是同一 period_number 横向比——比如所有 cohort 的 period_1 留存率。

面试官可能挖坑:"早期 cohort 的 LTV 更高,是不是用户质量在下降?"标准回答:"看同一 period_1 的留存率,各 cohort 都在 0.4%-0.7% 之间,用户质量是稳定的。final_ltv 的差异是观测窗口长度不同造成的机械效应。"

### Q4: 你怎么看 LTV 只涨了 3-5% 这个结果?

A: 这个结论比"LTV 增长了多少"更有价值的是"**量化了复购的上限**"。

具体含义:
- 首单客单价 ~160 BRL,最终 LTV ~165-175 BRL
- 也就是说,即使用户在平台上待了 17 个月,人均也只多贡献了 5-10 BRL
- 这意味着:**如果做推荐/召回/促销来提升复购,margin 非常薄** —— 花 10 BRL 营销成本去召回一个用户,可能连本都回不来

这反过来说明 Olist 当前阶段的正确策略确实是拉新而非留存——在复购率不到 1% 的市场里,花钱做留存 ROI 极低。

但面试中要学会转折:**"但这也意味着天花板很高——如果把次月留存从 0.5% 提升到 2%,LTV 就能翻倍。这就是推荐系统在 Olist 的业务价值。"**

### Q5: Cohort 分析的常见坑有哪些?

A: 四个坑:

**坑 1:口径不一致**。RFM 用 customer_unique_id,Cohort 必须也用;RFM 用 delivered 订单,Cohort 也必须用。口径不一致 = 两张图各说各话,面试直接翻车。

**坑 2:窗口效应**。早期 cohort 窗口长,LTV 自然高——不是用户质量更好,只是时间更多。按同一 period 横向对比才是正确做法。

**坑 3:数据截断**。最后一个 cohort(2018-08)的 period_1 只有 0.02%,不是用户质量差,是因为数据集在 2018-08-29 截断了,次月观测窗口不完整。不排除会被面试官质疑数据质量。

**坑 4:PERIOD_DIFF 格式**。MySQL 的 PERIOD_DIFF 需要 YYYYMM 格式的数值参数,传 '2017-03' 这种带横杠的字符串会计算错误。细节决定成败。

**坑 5:`v_orders_clean` 不是"干净"的**。视图只排除了 23 条时间异常订单,没有过滤订单状态(canceled、unavailable 等)。实际使用时仍需加 `WHERE order_status = 'delivered'`,否则未完成订单会被算进用户首购和复购统计中。教训:视图只能解决它设计时针对的问题,不能替代自己对口径的审查。

## Part 8: 品类经营分析

### Q1: 品类分析为什么要和 RFM/Cohort 串联？

A: RFM 回答了"谁在买"，Cohort 回答了"什么时候不买了"，品类分析回答"买什么导致了不再买"。

具体串联路径：
- RFM 品类首单分析 → bed_bath_table 是 8,432 人首单品类，评分 3.91 → 最大流失源
- Cohort → 整体留存极低，LTV 几乎不涨
- 品类分析 → bed_bath_table 评分确实低、履约确实慢，但同品类复购率 15.6% 也不算低

三个分析合在一起才构成完整的叙事：**用户来了 → 买了 bed_bath_table → 体验不好 → 不再回来 → LTV 停止增长。** 单看任何一个分析都不完整。

### Q2: office_furniture 到底是品类问题还是卖家问题？

A: 这个问题本身就是面试得分点——"不只看品类层面，自动钻取卖家维度验证假设"。

品类层面的数据：
- 评分 3.52（全品类倒数）、履约 20.79 天（全品类最慢）、差评率 25.44%
- 直觉告诉我：大件家具，天然慢，没办法。

卖家层面钻取后反转：
- 70% 订单来自一个卖家（id: 7c67e14...），评分 3.39、备货 11.47 天、履约 22.43 天
- 品类里有两个优质卖家：评分 4.18、履约 11.96 天（比劣质卖家快 10 天）

结论：**不是家具天然慢，是流量集中给了一个烂卖家。** 治理动作不是"优化品类"，而是"优质卖家加权推荐 + 劣质卖家限流"。这就是从分析到策略的完整闭环。

如果面试官追问"你怎么知道优质卖家产能足够？"可以回答：目前只拿到 12% 订单，如果给更多流量，需要监控 unavailable 率和备货时长是否恶化。但至少方向是对的。

### Q3: 同品类复购率差异这么大，怎么用到推荐系统？

A:

| 复购类型 | 代表品类 | 同品类复购率 | 推荐策略 |
|---|---|---|---|
| 功能型（同品类复购） | furniture_decor, signaling_and_security | 15-24% | 同品类召回：买了 bed_bath_table 推 bed_bath_table |
| 冲动型（跨品类复购） | cool_stuff, art, audio | 1-5% | 跨品类推荐：买了 cool_stuff 推 toys/stationery |

规则很简单：**首购品类同品类复购率 > 15% → 同品类召回优先；< 5% → 跨品类推荐优先；5-15% → 混合。**

面试官如果追问"怎么实现"，回答：在推荐系统的召回阶段，加一路"品类策略召回"——根据用户首购品类，动态选择同品类热门还是跨品类热门作为候选池。这是规则系统 + 模型的混合架构。

### Q4: bed_bath_table 到底该怎么治理？

A: 只有 15.6% 同品类复购率——不是最低的，但考虑到它是 8,727 首单用户的最大入口，绝对流失人数是最多的。

三管齐下：
1. **卖家分层**：按评分和履约把 bed_bath_table 的卖家分成 A/B/C 三档，A 档加权推荐，C 档降权或下架
2. **物流预期管理**：床浴室用品履约 12.71 天（品类均值），在商品详情页标注"预计 XX 天送达"，降低等待焦虑 → 减少差评
3. **交叉推荐**：首单 bed_bath_table 的用户，下个月推 housewares（评分 4.11/履约 10.87 天），体验好的相邻品类可以制造复购

### Q5: NULL category 1.31% GMV 你怎么处理？

A: 不是删除，而是标注。

- 1.31% 的 GMV 占比不影响全局结论
- 在报告和数据字典中标注"610 个产品缺少品类信息，GMV 占比 1.31%"
- 建议产品团队补全这些产品的品类标签
- 在推荐系统中，这些商品无法通过品类召回路径触达 → 只能靠热门召回和 ItemCF 召回覆盖

面试核心：**数据不全不是问题，不承认数据不全是问题。** 明确标注数据质量边界，反而体现分析严谨性。

## Part 9: 卖家健康度分析

### Q1: 你怎么构建卖家健康分？为什么不用 NTILE？

A: 健康分 = GMV分位(0-100) + 评分分位(0-100) + 备货时效分位(0-100) - 惩罚项(各-20)。

用 PERCENT_RANK 而非 NTILE 的原因：
- 卖家有 1,783 个（过滤后），数据分布连续，NTILE 可以工作
- 但 PERCENT_RANK 给出的是连续排名（0 到 1 之间的精确值），比 NTILE 的离散桶（1-5）更精细
- 惩罚项的阈值来自业务判断：缺货率 > 5%（每 20 单缺 1 单）、取消率 > 10%、差评率 > 25%

面试中如果被追问"权重合理吗"：坦诚说这是简化版，真实场景会用 AB 实验或层次分析法(AHP)校准权重。关键是你知道权重是业务选择而非数学真理。

### Q2: S 级卖家和只按 GMV 排名的 Top 卖家有什么区别？

A: 区别很大。

- GMV 最大的卖家（1,159 单/16.3 万）健康分只排第 93 → 量大但体验一般（评分 4.27/差评率 10%）
- 健康分第 1 的卖家 GMV 只有 16.8K → 量不大但体验极好（评分 4.71/备货 0.57 天）

如果平台只按 GMV 分配流量，就是在奖励差体验。健康分体系的核心价值：**在 GMV 和体验之间找到平衡点，让流量流动到"既卖得动又不让用户失望"的卖家。**

### Q3: office_furniture 那个烂卖家为什么没进 Bottom 10？

A: 这是一个很好的面试追问点——它暴露了当前健康分的一个设计偏差。

那个卖家（7c67e14...）891 单/22 万 GMV，GMV 分位接近 100。即使评分 3.39、履约 22 天、差评率 27.84%（触发了差评惩罚 -20），GMV 分位 + 评分分位 + 备货分位 = ~230 - 20 = ~210，仍然远高于 Bottom 10 的小卖家。

这说明：**当前 GMV 权重 1/3 可能过高。如果平台目标是提升用户体验，应该降低 GMV 权重或提高体验惩罚力度。权重选择本身就是一种业务表态。**

追问"你怎么确定最优权重"：回答"需要 AB 实验——控制组按 GMV 分配流量，实验组按健康分分配流量，看两组在 GMV/复购率/差评率上的差异。数据会告诉我们应该给体验多少权重。"

### Q4: 卖家集中度 46.44%（Top 100），这意味着什么？

A: 比品类集中度温和——品类 Top 5 占 39.24%，卖家 Top 50 才占 33.73%。同一品类内有多个卖家在竞争，供给多样性不错。

对推荐系统的含义：可以用卖家健康分做候选集过滤——在召回阶段过滤掉 D 级卖家，在排序阶段给 S/A 级卖家加权。不会因为过滤导致供给不足。

对平台的含义：5K+ GMV 的卖家贡献 86.09% GMV，小卖家虽然数量多但 GMV 贡献很低。平台策略应该是"保住中大型卖家体验 + 扶持小卖家中健康度高的"，而不是一刀切。

### Q5: D 级卖家全是小卖家，那卖家健康分还有用吗？

A: 有用，但作用层级不同。

- D 级小卖家：健康分告诉他们"你的体验数据还不足以评估"（valid_orders < 5 直接被过滤）。平台不需要惩罚他们，也不需要奖励。
- A/B 级中等卖家：这是健康分最有用的区间——GMV 有一定规模，但体验可能有分化。比如 office_furniture 的烂卖家就在这个区间，如果健康分能把它从 A 级拉到 B 级，流量分配系统就会相应调整。
- S 级卖家：健康分确认他们值得更多流量——这是给推荐系统和搜索排序的明确信号。

**健康分的核心用户不是 D 级，而是 A/B 级——那些 GMV 够大、体验有分化的卖家。**

## Part 10: 策略收益估算

### Q1: 你估算的 GMV 损失是怎么算出来的？有什么局限？

A: 两笔账。

**canceled 订单**：461 单，105,886 BRL。这些订单有完整的 order_items 数据，直接用 `SUM(price + freight_value)` 算。

**unavailable 订单**：数据只显示 6 单/2,140 BRL，但之前发现全平台有 609 单 unavailable。原因是 603 单缺货订单在数据集中没有 order_items（缺货 = 商品从未发出 = 没有 item 行）。我用替代估算：609 × 平台客单价均值 160 BRL ≈ 97,000 BRL。

**总 GMV 损失天花板 ~20 万 BRL，占全平台 1.3%。**

局限：主动说明——"unavailable 的估算用的是平台均值而非品类均值，实际数字可能偏高或偏低 15%。但量级是对的（在 10 万量级而非 100 万量级）。"

面试中主动提数据质量限制 = 诚实 = 加分。掩盖数据问题 = 面试翻车。

### Q2: 为什么风险卖家治理只找到一个卖家？是不是方法有问题？

A: 不是方法问题，是数据本身的限制。而且钻取后发现比"只有一个卖家"更根本。

609 单 unavailable 订单中，**603 单在 order_items 表里没有行**——缺货意味着商品从未发出，自然没有商品行。没有 order_items → 没有 seller_id → 无法归因到具体卖家。

排除无法归因的订单后：全平台只有 1 个卖家触发了 > 5% 的缺货门槛，且只有 1 单缺货（5 单中 1 单）。统计上完全没有意义。

三层结论：
1. **数据层**：Olist 的缺货数据不够完整，603/609 无法做卖家归因
2. **业务层**：可归因的少量数据暗示缺货是长尾问题而非集中问题
3. **策略层**：如果要系统治理缺货，先要完善订单-商品-卖家的数据链路

面试核心：**不是每个分析都能得出"治理方案"。有些分析的结论就是"数据不支持这个方向，需要先修数据"。敢说"我做不了"比硬编一个方案强得多。**

### Q3: 备货优化 0.5 天影响大吗？

A: 绝对量很大——SP 州 17,194 个慢订单，累计节省 8,597 天用户等待。

但有两个 caveat：
1. 0.5 天对用户感知的改善可能不明显（用户很难感知到 2.5 天 vs 3 天的差异）
2. 这些"慢订单"多数不是因为卖家懒，而是产品本身就是定制/大件（品类分析中 office_furniture 备货 10.16 天）

所以备货优化的核心不是"所有订单都快 0.5 天"，而是在商品详情页标注准确的预计备货时长——**让用户预期和实际匹配，比单纯加速更重要。**

### Q4: 早评偏差消除后评分提升 0.13 分，重要吗？

A: 0.13 分在 5 分制中是有意义的——相当于差评率降低 2-3 个百分点。

但要区分两件事：
- **消除测量偏差**：早评 2.79 分不是"用户体验差"，而是用户的评价还没包含产品体验。调整评价邀请时机（从发货后 → 签收后），这个偏差会自然消失 → +0.13
- **真正提升体验**：通过改善配送速度、产品质量来提升评分 → 这是另一个话题

面试中被追问"0.13 能带来多少 GMV 提升"：坦诚说评分和 GMV 之间没有线性公式。需要 AB 实验：控制组不变，实验组只显示签收后评价，看 CVR 有无差异。

### Q5: 这四个策略，优先级怎么排？

A:

| 优先级 | 策略 | 投入 | 收益 | 风险 |
|---|---|---|---|---|
| P0 | 早评机制调整 | 低（改通知逻辑） | 评分 +0.13 | 评论量可能短期下降 |
| P1 | canceled 治理 | 中（卖家管理） | 挽回 10.6 万 BRL | 可能误罚合理取消 |
| P2 | 备货预期管理 | 中（前端改动） | 体验改善（难量化） | — |
| P3 | unavailable 治理 | 高（库存系统） | 挽回 ~10 万 BRL | 小卖家缺乏技术能力 |

排序逻辑：**低投入高确定性的先做，高投入且效果不明确的用 AB 实验先验证。** 面试官看的不只是数字，更是你对业务优先级的判断力。

## Part 11: 推荐系统候选召回原型

### Q1: 你没有曝光/点击数据，怎么做推荐系统？

A: 我会在面试中主动说明这个边界——"Olist 没有曝光、点击、搜索词和加购数据，所以我构建的是基于购买行为的离线推荐候选召回原型，不是完整的工业推荐系统。"

但这不代表它没有价值：
- **样本构造**：从 SQL 清洗 → 事件级切分 → 训练/验证集分开，和工业流程完全一致
- **特征工程**：品类、评分、价格段、卖家健康度都可以作为排序特征
- **召回 baseline**：热门 + 同品类 + ItemCF 三条路径，用 Recall@K/NDCG@K 评估
- **冷启动意识**：主动分析用户/商品冷启动率

工业推荐系统 = 召回 → 粗排 → 精排 → 重排。我只做了一个环节（召回），但每个环节的数据管道是一样的。面试官看的是你知不知道推荐系统的数据是怎么来的，而不是你会不会调 sklearn。

### Q2: 为什么训练/验证集要用时间切分而不是随机切分？

A: 因为推荐系统是预测未来。

随机切分 = 把未来的数据混入训练集 = "用明天预测明天" = 信息泄漏。你训练的模型会"记住"用户买过什么，而不是"预测"用户会买什么。

时间切分 = 用 2018-06-01 之前的行为预测之后的行为 = 模拟真实线上场景。这样评估出来的 Recall@K 才是可信的下界。

面试中如果被问"随机切分和时间切分有什么区别"，核心答案就是八个字：**随机切分高估，时间切分真实。**

### Q3: 为什么商品特征只能用训练期数据？

A: 同样是因为时间泄漏。

举个例子：一台 iPhone 在 2018-08 大卖，销量暴涨。如果在训练商品特征时用了全量数据（含 2018-08），训练出的模型就知道"这台 iPhone 很热门"。但真实场景下，2018-06 之前你根本不知道这台 iPhone 会火。

训练期特征 = 2018-06-01 前的销量、评分、价格 → 模拟"模型在那一刻知道的全部信息"。

全量特征 = 作弊。热门 baseline 的 Recall 会比你想象的高很多，因为"未来销量"帮你选了答案。

### Q4: Olist 没有商品级评分，你怎么处理？

A: Olist 的 order_reviews 是订单级评价——一个订单一个评分，但一个订单可能包含多个商品。

我在 SQL 中先按 order_id 聚合（`AVG(review_score)`），再 JOIN 到商品上。相当于用订单评分近似商品体验评分。这不是完美的，但我在注释中主动标注了"Olist 无商品级评分，此处用订单评分近似"。

如果面试官追问"这不会引入偏差吗"：会。如果一单买了高评商品 + 低评商品，订单评分会平均掉两者的差异。在工业推荐中，商品级评分通常来自用户对单个商品的评价，这边确实做不到。但这不影响召回——召回阶段主要用协同信号（谁买了什么），评分更多是排序特征。

### Q5: 96.9% 用户冷启动，ItemCF 还有意义吗？

A: 有，但对于冷启动用户——Recall 会非常低。

冷启动用户没有历史行为 → ItemCF 找不到相似用户 → 无法做协同过滤。这类用户只能靠热门召回和品类召回覆盖。

但 ItemCF 的价值在**非冷启动用户**（~3,000 人，复购用户）。这些用户有历史购买 → 可以计算用户相似度 → ItemCF 的召回会比热门更个性化、更精准。

面试中正确的说法："ItemCF 的 Recall@K 在冷启动用户上很低（预测为 0），但在复购用户上应该显著优于热门召回。这恰恰验证了推荐系统在 Olist 的 ROI——如果大部分用户不复购，冷启动问题比协同过滤更重要。在 Olist 上，热门 + 品类召回 + 冷启动策略（新用户推高分快配送品类）比 ItemCF 更实用。"

### Q6: 你为什么要拆成 interaction_events → train/val 两张表？

A: 因为第一版的"先聚合再切分"有致命 bug。

第一版先把同一个 user_id + product_id 的所有购买聚合成一行，用 `last_purchase_time` 切 train/val。问题：如果一个用户 2018-05 买了 A、2018-07 又买了 A，由于 `last_purchase_time = 2018-07 >= 2018-06-01`，这条 pair 整个进入验证集，训练集里反而不存在这个用户的购买历史。

修正后：先保留事件级明细（每笔订单一行），按 `order_purchase_timestamp` 切分，再分别聚合。这样 2018-05 的订单在训练集、2018-07 的在验证集，互不污染。

**面试考点：推荐系统中最怕的不是模型不好，而是数据 pipeline 有信息泄漏——你在 SQL 层就把问题搞定了，比调模型参数重要得多。**

## Part 12: Python 召回 Baseline 评估

### Q1: ItemCF 的 Recall 全是 0，是不是代码写错了？

A: 不是。ItemCF 在这个数据集上就是不可用的——但这不是失败，而是整个项目最重要的结论之一。

数字链路：
- 商品共现对（原始）：11,940 对 → 过滤后（min_cooccur=2）：422 对 → 过滤了 96.5%
- 每个商品平均只有 1.3 个相似邻居 → 远不够覆盖验证集的 9,890 个商品
- Recall@K 全为 0

根因：96.9% 用户只买一次 → 两个商品被同一用户购买的概率极低 → 共现矩阵接近空集。

面试讲法："我不只是在 Olist 上跑了 ItemCF，我是用数据证明了 ItemCF 在 Olist 上不可行。11,940 个原始共现对被 min_cooccur=2 过滤掉 96.5% 后只剩 422 对——这本身就是结论。如果面试官让我做推荐系统，我会先基于这个发现建议用冷启动策略（热门 + 品类 + 评分/履约特征）而不是协同过滤。"

### Q2: GlobalHot 为什么是最强 baseline？

A: 因为 Olist 是一个获取型平台，97.72% 的验证集用户是冷启动的。GlobalHot 不需要任何用户历史，对冷启动用户的 Recall 和非冷启动一样。

| 用户群 | GlobalHot Recall@50 |
|---|---|
| 所有用户 | 3.73% |
| 冷启动（97.72%） | 3.75% |
| 非冷启动（2.28%） | 2.70% |

冷启动用户的 Recall 反而比非冷启动略高——因为冷启动用户平均验证集购买数更少（1.05 vs 1.13），分母更小。

**面试讲法**："在 Olist 上，一个连用户历史都不需要的 GlobalHot baseline 就打败了 ItemCF。这不说明 ItemCF 不好，而是数据模式决定了协同过滤没有信号可捕捉。面试官如果问你怎么在 Olist 上做推荐，正确的回答是'冷启动策略 + 热门 + 品类 + 卖家健康度过滤'，而不是'上深度学习模型'。"

### Q3: 为什么要区分 all_targets 和 warm_targets？

A: 两套口径回答不同的问题。

- **all_targets**：模拟真实线上场景——验证集里有什么就评估什么，包括训练期从未出现过的商品。这是"用户真正能感知到的推荐效果"。
- **warm_targets**：排除商品冷启动的噪声——只看训练期已有过的商品。这是"模型自身的召回能力"。

warm_targets 下 GlobalHot Recall@50 = 6.49%，all_targets 下 = 3.73%。商品冷启动拉低了近一半的 Recall。但 warm_targets 本身是理论天花板——真实场景不可能排除冷启动商品。

**面试讲法**："双口径评估是推荐系统论文里的标准做法。all_targets = 真实线上效果，warm_targets = 模型自身能力。两者之间的差距就是你给冷启动策略留的空间。"

### Q4: avg_recall_count 揭示了什么？

A: 看 CategoryHot 的 avg_recall_count：
- 非冷启动用户 @K=50：只召回了 10.3 个候选
- 设置 K=50 → 实际只能出 10 个 → 不是因为 K 不够，是因为用户的品类历史有限

这告诉我们的不是"把 K 调大就行"，而是"品类召回的天然瓶颈是用户品类的广度"。

### Q5: ItemCF 共现对只有 422 对，这个数字在工业界正常吗？

A: 完全异常。工业推荐系统（淘宝、京东、Shopee）的 item-item 共现对通常在百万到十亿量级。Olist 的 422 对是极端稀疏的。

但这恰恰是 Olist 项目的价值——它展示了**数据稀缺场景下推荐系统的务实做法**：不要死磕协同过滤，用规则召回（热门、品类、评分、履约）+ 冷启动策略。大厂的推荐系统也有冷启动场景（新用户、新商品），Olist 只不过是把"冷启动场景"放大到了 97.72%。处理冷启动的能力，在任何公司都是通用的。

## Part 13: 数据分析岗专项 — 指标体系 × 归因 × 看板

### Q1: 你怎么给 Olist 设计指标体系？

A: 三层结构——北极星 → 一级 → 二级 → 三级诊断。

```
北极星指标: GMV（或有效成交订单数，取决于平台当前阶段）

一级指标（回答"平台健康吗"）:
├── 用户规模: DAU/MAU、新增用户数、累计用户数
├── 成交转化: 下单转化率、付款转化率、成交率
├── 客单价: AOV = GMV / 订单数
├── 复购: 复购率（次月留存、3 月留存）
├── 履约: 备货时长、配送时长、取消率、缺货率
├── 评价体验: 平均评分、差评率、早评率
└── 卖家供给: 活跃卖家数、卖家健康分均值

二级指标（回答"哪个环节出问题了"）:
订单数 = 活跃用户数 × 人均订单数
GMV = 订单数 × AOV = 用户数 × 人均订单数 × (商品均价 + 运费)

三级诊断指标（按维度拆分）:
├── 按时间: 月/周趋势
├── 按地域: 各州 × 指标
├── 按品类: 各品类 × 指标
├── 按卖家: S/A/B/C/D 级 × 指标
└── 按用户: 新用户 vs 老用户 × 指标
```

面试讲法："指标体系不是指标列表——是有层次、有公式、有下钻路径的。上级指标异常 → 公式拆解 → 下钻到时间/地域/品类找根因。这套方法在 Olist 上练过，到任何电商平台都一样。"

### Q2: GMV 涨了/跌了，你怎么归因？

A: 多维归因框架——先拆公式，再下钻维度。

```
第一步: 公式拆解
GMV = 订单量 × 客单价
    = (活跃用户数 × 人均订单数) × (商品均价 + 运费)

第二步: 计算各因子的贡献度
ΔGMV = Δ订单量 × 基期AOV + ΔAOV × 基期订单量 + Δ订单量 × ΔAOV（交叉项）

第三步: 下钻到维度找根因
- 如果订单量跌 → 是新用户少了还是老用户少了？哪个品类的订单跌了？
- 如果 AOV 跌 → 是商品价跌了还是运费少了？是低客单品类占比变大了还是同类商品在降价？

第四步: 排除数据噪声
- 季节效应（黑五、圣诞节）
- 数据截断（2018-08 只有半个月数据）
- 异常事件（609 单缺货集中在某月）
```

以 Olist 为例：2017-11 黑五 GMV 环比 +53.55%，订单量环比 +62.77%，AOV 环比 -5.66%。增长主要来自"量"而非"价"，更具体地说是购买用户数增长驱动，而不是老用户购买频次提升。

面试讲法："这类促销增长不能只看 GMV 涨了，还要看是不是靠降价换来的。Olist 这里是订单量大涨、AOV 下滑，所以我会把它定义为促销拉新驱动型增长。"

### Q3: 你怎么设计一个监控看板？

A: 分层看板 + 异常预警规则。

四个看板：

| 看板 | 核心指标 | 刷新频率 |
|---|---|---|
| 经营总览 | GMV、订单量、AOV、用户数、复购率 | 日级 |
| 履约看板 | 付款耗时、备货耗时、配送耗时、取消率、缺货率 | 日级 |
| 品类看板 | 品类 GMV TOP 10、评分、差评率、配送时长 | 周级 |
| 卖家看板 | 卖家健康分分布、S 级/D 级卖家数、缺货率 | 周级 |

异常预警规则：

```
GMV 环比下降 > 10% → 自动告警
某品类差评率 > 全站均值 × 2 → 标记关注
卖家 unavailable 率 > 5% → 邮件通知运营
配送时长 > 历史 P90 → 物流团队沟通
月度复购率下降 > 20% → 用户运营复盘
```

面试讲法："看板不是图好看——是帮助业务方 3 秒内判断'平台今天健康吗'，1 分钟内定位'哪个环节出问题'。异常规则是自动化的'数字哨兵'，没人会天天盯着 20 个图表看。"

### Q4: 为什么北极星指标选 GMV，而不是评分、复购率或订单数？

A: 因为 Olist 是电商交易平台，最核心的商业结果还是成交规模，GMV 能直接反映平台撮合买卖双方的能力。

但我不会只看 GMV。我会把 GMV 作为北极星，同时设置一组护栏指标：

| 指标 | 作用 |
|---|---|
| 有效成交订单数 | 防止 GMV 被少数高客单订单拉动 |
| 取消率 / 缺货率 | 防止平台为了 GMV 忽视失败订单 |
| 平均履约时长 | 防止成交后体验变差 |
| 差评率 / 早评率 | 防止评分被系统机制污染 |
| 卖家健康分 | 防止流量过度奖励高 GMV 但低体验卖家 |

面试讲法："北极星指标负责定义业务方向，护栏指标负责防止方向跑偏。GMV 是结果，但履约、评价、缺货、取消决定这个 GMV 能不能长期持续。"

### Q5: 指标异常时，你怎么从指标体系定位根因？

A: 我会按"公式拆解 + 维度下钻"两步走。

第一步先拆公式：

```text
GMV = 订单量 × 客单价
    = 活跃购买用户数 × 人均订单数 × 客单价
```

如果 GMV 下降，先判断是订单量下降还是客单价下降。如果是订单量下降，再拆成用户数下降还是人均订单数下降。如果是客单价下降，再看商品价格、运费、品类结构是否变化。

第二步再下钻维度：

```text
时间: 是长期趋势还是节假日波动
地域: 哪些州下降
品类: 哪些品类拖累
卖家: 是否集中在少数卖家
用户: 新用户还是老用户变化
履约: 是否备货或物流变慢
评价: 是否差评率或早评率异常
```

以 Olist 为例，office_furniture 表面上是品类评分低、履约慢，但继续下钻到卖家后发现 70% 订单集中在一个低评分慢履约卖家手里。这个案例说明，指标异常不能停在一级维度，必须继续下钻到可行动的责任对象。

面试讲法："定位根因的终点不是'哪个指标变了'，而是'谁能采取行动'。如果下钻到品类还不能行动，就继续下钻到卖家；如果下钻到州还不能行动，就继续拆履约、供给和用户结构。"

## Part 14: AB 实验设计

### Q1: 你为什么要做 AB 实验，前面的分析不是已经给出结论了吗？

A: 前面的分析给出的是相关性和策略假设，不是因果结论。

比如早评分析显示收货前评价分数显著低于收货后，消除早评偏差后平台评分理论上可以提升 0.13 分。但这并不等于"改评价邀请时机一定能提升业务"。它可能带来副作用，比如评论提交率下降、评论延迟变长。

所以我的方法是：

```text
离线分析发现问题
-> 提出策略假设
-> 设计 AB 实验
-> 同时观察核心指标和护栏指标
-> 决定是否上线
```

面试讲法："分析给出假设，AB 实验验证因果。没有实验分组，就不能把相关性说成因果。"

### Q2: 评价邀请时机实验怎么设计？

A: 控制组维持发货后发送评价邀请，实验组改为签收后发送。

| 项目 | 设计 |
|---|---|
| 实验对象 | delivered 订单用户 |
| 控制组 | 发货后发送评价邀请 |
| 实验组 | 签收后发送评价邀请 |
| 核心指标 | 平均评分、差评率、早评率 |
| 护栏指标 | 评论提交率、评论延迟天数、客服投诉率 |

预期结果是早评率下降、平均评分上升、差评率下降。但关键是护栏指标：如果评分上升但评论提交率大幅下降，说明我们只是让不满用户不评论了，评分信号反而变得更偏。

面试讲法："这个实验的目标不是单纯把评分做高，而是让评分更准确地反映收货后的真实体验。"

### Q3: 高健康度卖家加权实验怎么设计？

A: 控制组使用原排序逻辑，实验组对 S/A 级卖家商品适度加权，对 D 级卖家轻度降权。

核心指标看 GMV、CVR、订单量，因为平台不能为了体验完全牺牲成交。护栏指标看平均评分、差评率、取消率、缺货率、卖家覆盖率。

为什么要看卖家覆盖率：如果实验组只把流量集中给少数 S 级卖家，短期体验可能更好，但长期会伤害供给多样性，小卖家没有成长机会。

面试讲法："卖家健康分不是为了替代 GMV，而是给 GMV 加一个体验约束。实验要验证的是'在 GMV 不显著下降的前提下，体验能否提升'。"

### Q4: 缺货高风险供给降权实验有什么数据限制？

A: Olist 这个数据集对缺货治理有天然限制：609 单 unavailable 中，603 单没有 order_items，因此没有 product_id 和 seller_id，无法精确归因到商品或卖家。

所以我不会在离线数据里硬编"哪个卖家缺货最多"。正确做法是：

```text
第一步:承认当前数据链路无法支持精确卖家级缺货治理
第二步:在线上补齐 unavailable 订单的商品和卖家归因
第三步:再设计高风险供给降权实验
```

实验上线后，核心指标看 unavailable 率和 canceled 率，护栏指标看 GMV、订单量、商品覆盖率、卖家覆盖率。

面试讲法："有些分析的结论不是'马上治理谁'，而是'当前数据还不能支持这个治理动作，必须先补数据链路'。这比硬给一个不可靠方案更专业。"

### Q5: 冷启动推荐实验为什么比 ItemCF 更值得做？

A: 因为 Olist 验证期用户冷启动率高达 97.72%，ItemCF 在评估中 Recall@K 全为 0。问题不是模型调参，而是用户历史行为太少，共现矩阵没有足够信号。

因此更值得做的实验是冷启动高质量热门推荐：

| 项目 | 设计 |
|---|---|
| 实验对象 | 新用户 / 无历史购买用户 |
| 控制组 | 纯 GlobalHot |
| 实验组 | QualityGlobalHot = 热门 + 高评分 + 快履约 + 高健康卖家 |
| 核心指标 | CVR、首单 GMV |
| 护栏指标 | 首单评分、差评率、配送时长、商品覆盖率 |

这个实验更贴近 Olist 的真实业务，因为绝大多数用户没有历史行为。与其在极稀疏数据上死磕 ItemCF，不如先把冷启动首页推荐做好。

面试讲法："推荐系统不是越复杂越好，而是要和数据形态匹配。Olist 的主要矛盾是冷启动，不是协同过滤相似度不够精细。"

## Part 15: VOC 评价文本分析

### Q1: 为什么已经有评分了，还要做评论文本分析？

A: 评分只能告诉我"用户不满意"，但不能告诉我"为什么不满意"。

比如同样是 1 星，可能来自完全不同原因：

```text
物流慢
没收到货
商品破损
质量不符合预期
退款/客服体验差
```

这些原因对应的业务动作完全不同。物流慢要找履约团队，商品破损要找卖家和包装，退款客服差要找客服流程。VOC 的价值就是把一个抽象的低评分拆成可行动的问题类型。

在 Olist 里，1 星评论文本覆盖率是 76.55%，明显高于 5 星的 35.84%。差评文本中物流配送问题占 21.98%，未收到货占 16.96%，两者合计 38.94%，说明低分的主要可解释原因来自履约链路。

面试讲法："评分是结果，文本是原因。数据分析不能只告诉业务'分低了'，还要告诉业务'为什么低、谁来改'。"

### Q2: 你为什么用关键词规则，而不是直接上 NLP 模型？

A: 因为这个项目的目标是 SQL 数据分析，不是 NLP 建模。关键词规则有三个优点：

| 优点 | 说明 |
|---|---|
| 可解释 | 每个分类都能看到触发词 |
| 可复现 | SQL 规则固定，业务方能审核 |
| 低成本 | 不需要训练模型，适合快速定位问题 |

缺点是召回不完整，无法理解复杂语义、讽刺、同义词和拼写错误。所以我会主动标注：这是方向性 VOC 分析，不是完整 NLP 情感模型。

面试讲法："先用简单可解释规则验证问题是否存在。如果业务价值足够大，再升级到分词、主题模型或大模型分类。不要为了炫技一开始就上复杂模型。"

### Q3: VOC 分析和前面的早评/地域分析有什么关系？

A: 前面的早评分析发现：收货前评价评分明显低于收货后，且不同州早评率差异很大。

VOC 可以进一步验证这些低分到底在抱怨什么。如果早评低分里大量关键词是 `entrega`、`atraso`、`demora`，说明用户不是在评价商品质量，而是在等待物流时表达不满。

这能支撑一个很重要的结论：

```text
Olist 的评分信号混入了物流体验，
如果推荐系统直接用评分当商品质量 label，
会把物流慢的商品或地区系统性降权。
```

面试讲法："VOC 是对早评偏差的语义验证。数值上看到评分低，文本上看到用户在骂物流，两者互相印证。"

### Q4: VOC 分析有哪些数据局限？

A: 主要有三类局限。

第一，评论文本不是全量。很多用户只打分不留言，所以文本分析只代表愿意写评论的人，存在选择偏差。不过这个选择偏差本身也有业务含义：Olist 里 1 星文本覆盖率 76.55%，5 星只有 35.84%，说明强不满用户更愿意写出具体原因。

第二，Olist 是订单级评价，不是商品级评价。一单多商品时，用户评论可能针对其中一个商品，但我们只能归因到整个订单或品类。

第三，关键词规则有漏召回和误判。比如用户用同义表达、拼写错误或讽刺语气，SQL `LIKE` 不一定能识别。

面试讲法："VOC 分析是辅助诊断，不是最终因果结论。我会把它和评分、履约时长、品类、地域一起交叉验证，而不是单独下结论。"

### Q5: 如果 VOC 发现物流类差评最多，下一步怎么做？

A: 我会分三步。

第一步，量化影响：看物流类差评占所有差评的比例，以及它集中在哪些州、品类和卖家。

第二步，定位责任对象：如果集中在跨州线路，找物流网络；如果集中在某品类，检查大件/易损商品；如果集中在某卖家，纳入卖家健康治理。

第三步，设计策略和实验：比如延后评价邀请、优化预计送达时间展示、对慢履约卖家降权、给高风险线路增加履约提示。

Olist 的结果支持这个方向：21 天以上订单的物流投诉占比是 33.12%，而 <7 天订单只有 12.16%，几乎高出 2.7 倍。地域上，RJ 的未收到货占 22.94%，物流配送问题占 22.48%；AL 的物流配送问题占比也很高。

面试讲法："VOC 不是写完词频表就结束，而是要把文本主题映射到业务责任对象和策略动作。"

## Part 16: 推荐混合召回 Hybrid Baseline

### Q1: 为什么在 GlobalHot、CategoryHot、ItemCF 之外还要做 HybridRecall？

A: 因为三条单路召回各自有明显短板。

GlobalHot 对冷启动用户最稳，但不利用用户历史。CategoryHot 能利用历史品类，但冷启动用户没有历史，所以召回为空。ItemCF 在 Olist 上几乎失效，因为验证期用户冷启动率 97.72%，多数用户只买一次，共现矩阵太稀疏。

所以我做了 HybridRecall:

```text
冷启动用户: GlobalHot
非冷启动用户: CategoryHot 优先，不足用 GlobalHot 补足
```

面试讲法："HybridRecall 不是为了堆算法，而是把不同召回路放到它最适合的人群上。"

### Q2: HybridRecall 的效果比 GlobalHot 好在哪里？

A: 在全量验证集上，HybridRecall 的 Recall@50 是 3.75%，略高于 GlobalHot 的 3.73%。在 warm_targets 口径下，HybridRecall Recall@50 是 6.54%，也略高于 GlobalHot 的 6.49%。

更关键的是非冷启动 warm 用户:

```text
GlobalHot Recall@50 = 4.78%
HybridRecall Recall@50 = 6.57%
```

这说明 Hybrid 的提升主要来自有历史行为的用户。它先用 CategoryHot 捕捉轻个性化，再用 GlobalHot 补足候选，既不牺牲冷启动兜底，也能利用少量历史信号。

面试讲法："整体提升不大，是因为 97.72% 用户是冷启动；但在真正有历史的用户上，Hybrid 的提升是明显的。"

### Q3: QualityHybrid 为什么 Recall 反而低？

A: QualityHybrid 把 GlobalHot 按评分、差评率等体验指标重排，相当于更偏向"高质量首单体验"，而不是纯购买命中。

结果 all_targets Recall@50 从 GlobalHot 的 3.73% 降到 1.20%。这说明高评分商品不一定是未来最常被购买的商品，体验优先和购买命中之间存在 trade-off。

所以我不会把 QualityHybrid 当作主召回最优方案，而会把它放进 AB 实验:

```text
控制组: GlobalHot / HybridRecall
实验组: QualityHybrid
核心指标: CVR、首单 GMV
护栏指标: 首单评分、差评率、配送时长
```

面试讲法："QualityHybrid 的低 Recall 不是坏结果，它告诉我体验护栏会带来购买命中损失，需要通过 AB 实验判断是否值得。"

### Q4: ItemCF 失败后，你怎么避免项目显得像负结论？

A: 我会把 ItemCF 失败解释成数据诊断结果，而不是模型失败。

ItemCF 需要用户多次购买形成商品共现，但 Olist 96.9% 用户只买一次。我的评估结果是共现对 11,940，过滤后相似商品对只有 422，Recall@K 全为 0。这个结果说明当前数据不支持协同过滤。

然后我基于这个发现转向 HybridRecall:

```text
主要矛盾: 冷启动和低复购
策略选择: 热门兜底 + 历史品类轻个性化
后续方向: 用品类、卖家健康、评分、履约做排序特征
```

面试讲法："好的推荐系统不是盲目上复杂模型，而是先判断数据里有没有模型需要的信号。"

### Q5: 如果要把这个离线召回原型推进到线上，你下一步怎么做？

A: 我会分三步推进。

第一步，补数据链路。当前 Olist 没有曝光、点击、加购和实验分组，所以离线 Recall 只能验证购买命中，不能评估 CTR、CVR 和真实转化。

第二步，扩展召回和排序特征。召回层保留 HybridRecall，排序层加入品类、价格段、卖家健康分、评分、差评率、备货时长、配送时长等特征。

第三步，设计 AB 实验。控制组用 GlobalHot 或当前线上策略，实验组用 HybridRecall 或 QualityHybrid，核心指标看 CVR、GMV，护栏指标看差评率、履约时长、商品覆盖率和卖家覆盖率。

面试讲法："离线召回原型只是第一步，真正上线要补曝光点击链路、排序特征和 AB 实验闭环。"
