> 项目:Brazilian E-Commerce Public Dataset by Olist

## 一、项目背景

本报告记录 Olist 巴西电商数据集导入 MySQL 后的数据质量检查过程,覆盖**行数核对、主键唯一性、外键完整性、时间逻辑、业务异常值、空值分布**六个维度。

数据从 Kaggle 公开数据集导出,Olist 是巴西最大的电商平台之一,数据时间跨度 2016-2018,包含订单、商品、客户、卖家、地理位置、评价等核心维度。

**核心目标**:在做任何业务分析之前,先回答"哪些数据可信、哪些要打折扣、哪些必须排除"。

## 四、详细检查结果

### 4.1 行数最终核对

**目的**:验证 LOAD DATA / Python 导入零丢失。
**检查**:`SELECT 'table', COUNT(*) UNION ALL ...`
**结果**:9 张表实际行数与 Kaggle 官方数字严格一致,零丢失。

**踩坑记录**:
- **CSV 行尾**:Kaggle CSV 实际是 LF(`\n`),不是 Windows 风格的 CRLF。VSCode 状态栏显示 CRLF 是误导(显示是保存目标格式,不是当前字节)。用 hex (`xxd file.csv | head`) 看实际字节才能确认。
- **教训**:`LINES TERMINATED BY '\n'` 才是正确写法,初版写 `'\r\n'` 导致 customers 表 0 行入库。

### 4.2 主键唯一性

**目的**:LOAD DATA 不强制检查 PK 唯一性,需手动验证。
**写法**:对每张表执行子查询 + COUNT:

```sql
SELECT COUNT(*) FROM (
    SELECT pk_col FROM tbl GROUP BY pk_col HAVING COUNT(*) > 1
) t;
```

**结果**:9/9 表 `duplicate_groups = 0`,主键完全干净。

### 4.3 外键完整性(找孤儿)

**目的**:加外键约束前,确认所有子表引用都能在父表找到。
**写法**:`LEFT JOIN` + `IS NULL` 找孤儿。

| 引用关系 | 孤儿数 | 处理 |
|---|---|---|
| order_items → products | 0 | ✅ |
| order_items → sellers | 0 | ✅ |
| order_items → orders | 0 | ✅ |
| orders → customers | 0 | ✅ |
| order_payments → orders | 0 | ✅ |
| order_reviews → orders | 0 | ✅ |
| **products → category_translation** | **13** | 🟡 修复 |

**钻取分析**:13 条孤儿只涉及 **2 个未翻译类别**:
- `pc_gamer`(3 条)
- `portateis_cozinha_e_preparadores_de_alimentos`(10 条)

**决策**:**补维度,不删事实**。13 条都是真实商品有订单记录,删了影响下游分析。
**实施**:`05_fix_translation.sql`,补 2 条翻译,翻译表行数 71 → 73,孤儿数 13 → 0。

💡 **数据治理原则**:维度表 = 可补,事实表 = 不可改。这是工业界 DWH 的铁律。

⚠️ **NULL 不是孤儿**:products 表中 610 条 `product_category_name IS NULL` 不算孤儿(没有引用值,不参与外键检查)。这是 SQL 三值逻辑(TRUE/FALSE/UNKNOWN)的标准行为,MySQL 外键约束默认豁免 NULL。

### 4.4 时间逻辑检查

**目的**:检查时间字段是否符合物理因果(付款 < 发货 < 送达)。

| 检查项 | 异常数 | 状态 |
|---|---|---|
| 付款时间早于下单时间 | 0 | ✅ |
| 送达时间早于发货时间 | **23** | 🟡 标记排除 |
| 评价时间早于下单时间 | 0 | ✅ |

#### 4.4.1 23 条"送达早于发货"钻取

用 `TIMESTAMPDIFF(MINUTE, carrier, customer)` 计算时差,分布呈两极:

| days_diff 区间 | 大致数量 | 推测根因 |
|---|---|---|
| -16 ~ -4 天 | ~9 条 | **数据严重错乱**(物流回写延迟 / 录入错误 / 数据迁移错位) |
| -1 ~ -3 天 | ~7 条 | 时区错位 + 部分数据问题 |
| < -1 天(几小时差) | ~58 条 | **时区错位为主**(巴西横跨 4 个时区 UTC-2 ~ UTC-5) |

**最严重的 case**:`c1e2bf2b` 显示 carrier=2017-03-02,customer=2017-02-14,**早了 16 天**,绝不可能是时区(时区差最多 12 小时)。

**决策**:不修复(无法确定真实时间),创建 `v_orders_clean` 视图,后续配送时长分析时排除这些异常订单。

```sql
CREATE OR REPLACE VIEW v_orders_clean AS
SELECT *
FROM orders
WHERE NOT (order_delivered_customer_date < order_delivered_carrier_date)
   OR order_delivered_carrier_date IS NULL
   OR order_delivered_customer_date IS NULL;
```

💡 **工业界标准做法**:保留原始数据 + 提供清洗视图,两份并存。**不要"伪造"数据**(不知道真实时间就别瞎改)。

### 4.5 业务异常值检查

**结果总览**:

| 检查项 | 异常数 | 状态 |
|---|---|---|
| review_score 不在 [1,5] | 0 | ✅ |
| price <= 0 | 0 | ✅ |
| freight_value < 0 | 0 | ✅ |
| product_weight_g = 0 或 > 30000 | **5** | 🟡 钻取 |

#### 4.5.1 评分分布(业务洞察金子)

```
1 星: 11424  (11.5%)
2 星:  3151  ( 3.2%)
3 星:  8179  ( 8.2%)
4 星: 19142  (19.3%)
5 星: 57328  (57.8%)
```

**两个重要洞察**:
1. **5 星占 57.8%** —— 典型的"幸存者偏差":不满意的用户很多直接不评分
2. **U 型分布**:1 星(11.5%) + 5 星(57.8%) ≈ 70% 极端评价,中性评价(3 星)占比小

💡 **对推荐算法的影响**:不能直接用评分作回归 label,要做去偏处理(IPW / DR estimator)。这是工业界做 LTR / CVR 模型的核心问题。

#### 4.5.2 weight_g 异常 5 条钻取

**全部是同一类别 `cama_mesa_banho`(床上/餐桌/浴室用品)**。

- **4 条 weight = 0**:体积都是 22500 cm³(30×25×30,中等床品大小),真实重量应该 500g~2kg,**0 是录入缺失,默认值**
- **1 条 weight = 40425g (40.4kg)**:尺寸 13×65×28 cm,疑似单位错误或大件床品压缩包

**决策**:
- 4 条 weight = 0 → **改为 NULL**(规范化)
- 40.4kg 那条 → 暂保留,需关联订单确认是真重物还是单位错

```sql
UPDATE products
SET product_weight_g = NULL
WHERE product_weight_g = 0;
```

💡 **为什么 0 比 NULL 更糟**:0 是合法数字会被 `AVG()` 聚合算进去拉低均值,NULL 会被自动排除。这是数据分析师常见的"假数据陷阱"。

### 4.6 空值分布 + 订单生命周期

#### 4.6.1 第一次踩坑(LOAD DATA BUG)

初版 LOAD DATA 在 SET 子句前多了一个 `;`,导致 LOAD DATA 提前结束。结果:
- `order_id`, `customer_id`, `order_status` 等字段正常入库
- `order_approved_at`, `order_delivered_carrier_date`, `order_delivered_customer_date` 全部 NULL

**关键观察**:LOAD DATA **没报错**,行数也对(99441),只在数据质量检查时发现 `missing_delivered = 99441` 才反推回去定位 BUG。

💡 **教训**:LOAD DATA 是"看似成功的失败"经典场景。数据导入后**必须立刻做质量检查**,不能假定"没报错 = 成功"。工业界用 Great Expectations / dbt test 自动化这种校验。

#### 4.6.2 订单生命周期(按 status × NULL 分布)

修复 LOAD DATA 后重新跑,得到完整订单状态机:

| status | cnt | missing_delivered | missing_carrier | missing_approved | 业务解读 |
|---|---|---|---|---|---|
| delivered | 96478 | **8** ⚠️ | 2 | 14 | 已送达,本该完整,但有 8 单送达却无送达时间 |
| shipped | 1107 | 1107 | 0 | 0 | 已发货未送达,合理 |
| canceled | 625 | 619 | 550 | 141 | 取消可发生在任何时点 |
| unavailable | 609 | 609 | 609 | 0 | 商品缺货,已付款但未发货 |
| invoiced | 314 | 314 | 314 | 0 | 已开发票未发货 |
| processing | 301 | 301 | 301 | 0 | 处理中,等待发货 |
| created | 5 | 5 | 5 | 5 | 刚创建 |
| approved | 2 | 2 | 2 | 0 | 已批准未发货 |

**业务漏斗(整体)**:
- 未付款率:160/99441 = **0.16%**
- 未发货率:1783/99441 = **1.79%**
- 未送达率:2965/99441 = **2.98%**

漏斗逐层递增,符合电商订单状态机的典型损耗规律。

#### 4.6.3 🔴 8 单"幽灵送达"钻取(本次最深价值发现)

`delivered` 状态下 8 单缺 customer_date,钻取后:

- **7 单**有 carrier_date,1 单 carrier_date 也是 NULL
- **集群特征**:5 单 carrier_date 集中在 **2018-07-01 ~ 2018-07-03**(3 天内)
- **同卖家关联**:`2ebdfc4f` 和 `e69f75a7` 都是 seller `58f1a6197ed863543e0136b`,且 carrier 时间精确到秒一致(都是 `2018-07-03 13:57:00`)

**根因推断**:**卖家批量发货时物流回调丢失**。同一卖家同一时刻批量调用 API 发货,部分回调失败导致 customer_date 没回写,但订单 status 已被前置流程改成 delivered。

**对比另一个 case**:`2d858f45` 连 carrier_date 都是 NULL,可能是**纯手工改 status** —— 客服在 Olist 后台直接把 status 改成 delivered,绕过状态机(线下签收 / 关闭历史挂单 / 数据迁移遗漏)。

💡 **工程治理建议**:
1. **数据校验**:carrier_date 已填但 customer_date NULL 超过 30 天就告警
2. **状态机约束**:status=delivered 强制要求 customer_date NOT NULL(CHECK 约束或触发器)
3. **物流回调幂等性**:确保丢失的回写可以补传

💡 **数据分析师视角**:**异常不是均匀分布,是集群的**。8 单看似随机,深挖发现 5 单共享时间窗 + 2 单共享卖家。这种细节钻取能力是面试时讲故事的核心素材。

### 4.6.4 🔴 8320 条收货前评价:地域交叉分析(深度钻取)

#### 背景

在 4.6.2 订单生命周期分析中,发现 8320 条记录的 `review_creation_date < order_delivered_customer_date`(评论早于送达,占 8.6%)。4.4 节已确认这是 Olist 评价触发时机设计缺陷(发货即发邀请,而非签收后)。本节按 **customer_state(客户所在州)** 交叉分析,验证是否存在地域偏差。

#### SQL

```sql
SELECT 
    c.customer_state,
    COUNT(*) AS total_reviews,
    SUM(CASE WHEN r.review_creation_date < o.order_delivered_customer_date 
        THEN 1 ELSE 0 END) AS early_reviews,
    ROUND(AVG(CASE WHEN r.review_creation_date < o.order_delivered_customer_date 
        THEN r.review_score END), 2) AS early_avg_score,
    ROUND(AVG(CASE WHEN r.review_creation_date >= o.order_delivered_customer_date 
        THEN r.review_score END), 2) AS normal_avg_score
FROM orders o
JOIN order_reviews r ON o.order_id = r.order_id
JOIN customers c ON o.customer_id = c.customer_id
WHERE o.order_delivered_customer_date IS NOT NULL
GROUP BY c.customer_state
HAVING early_reviews > 30
ORDER BY early_avg_score ASC;
```

#### 发现 1:所有州无一例外,early_score < normal_score

21 个州**全部**呈现收货前评分低于收货后。gap 从 -0.92 (MT) 到 -2.22 (SE)。没有一个州反转。这证实了 4.4 节的结论:**这是系统性的 Olist 产品设计缺陷**,不是偶发数据错误。

#### 发现 2:"偏远 = 分低"假设被推翻(关键修正)

初始假设:偏远地区(亚马逊流域)物流慢 → 收货前评分更低。

**数据反驳**:
- **RJ(里约热内卢)排第三差**(early_score=2.24),但里约是巴西第二大城市、东南沿海经济中心,物流基础设施一流
- **MT(马托格罗索)排第二好**(early_score=3.31),但马托格罗索是内陆农业州,地广人稀,物流条件远不如 RJ

真正的规律不是"偏远 = 分低",而是看 **early%**(早评率):

#### 发现 3:早评率(early%)才是关键指标

| 早评率 | 代表州 | early_score | 解读 |
|---|---|---|---|
| >15% | AL(20.6%), MA(18.0%), CE(16.7%), SE(15.0%) | 2.02~2.44 | 高愤怒区 |
| 10~15% | RJ(13.5%), PA(13.7%), BA(13.4%) | 2.24~2.57 | 中等 |
| <10% | PR(6.2%), SP(6.6%), MG(6.7%), DF(7.5%), SC(7.7%) | 3.06~3.35 | 低愤怒区 |

**核心规律**: `early%` 越高 → `early_score` 越低。早评率高 = 该州用户在配送中收到评价邀请的比例高 = 配送时间长 = 等待中的愤怒。

#### 发现 4:真正的主变量是"期待差",而非"绝对配送时长"

```
RJ(大城市): 期待 3 天到 → 等了 7 天 → 给 2.24 分
MT(内陆):   期待 15 天到 → 等了 20 天 → 给 3.31 分
```

**不是物流慢不慢的问题,是"有没有超出用户预期"的问题**。RJ 用户对电商配送有高期待(隔壁 SP 一天就到),物流延迟超出预期后愤怒值高。MT 用户习惯了长配送周期,多等几天不觉得意外。

💡 **本质**:评价 = 体验 - 期待。Olist 产品设计让"物流体验"伪装成了"商品评价",而各州用户的期待基准不同,造成了**系统性但非均匀**的偏差。

#### 对推荐系统的影响

| 影响维度 | 具体后果 |
|---|---|
| **地域公平性** | AL/SE 等小州的好商品会被系统性地打低分 → 被推荐模型降权 → 这些州的卖家在平台上处于劣势 |

## 七、RFM 修正记录(2026-06-03)

### 7.1 初始执行的 3 个问题

第一版 RFM 直接对三个维度都用 `NTILE(5)`,跑完发现用户分层严重反常:

| 分层 | 错误版 avg_monetary | 应该有 | 问题 |
|---|---|---|---|
| 核心用户 | 56 雷亚尔 | 应最高 | NTILE 方向反了 |
| 流失用户 | 310 雷亚尔 | 应最低 | 同上 |

定位到 3 个根本原因:

**错误 1:R 维度 NTILE 方向反了**

`NTILE(5) OVER (ORDER BY recency_days ASC)` — 升序排列,recency=1 天(最好)排第一得 1 分,recency=900 天(最差)排最后得 5 分,方向反了。

修正:`ORDER BY recency_days DESC` — recency 小(最近购买)的排最后,拿到高分。

**错误 2:M 维度 NTILE 方向反了**

`NTILE(5) OVER (ORDER BY monetary DESC)` — 降序排列,最高消费排第一得 1 分,最低消费排最后得 5 分,方向反了。

修正:`ORDER BY monetary ASC` — 低消费排前面拿低分,高消费排后面拿高分。

**错误 3:F 维度 NTILE 在低复购数据上完全失效**

Olist 96.9% 用户 frequency=1,NTILE(5) 强行把 93,099 个同频用户切分到 5 个桶,等同于随机分组。

修正:改用业务阈值 `CASE WHEN frequency=1 THEN 1 WHEN frequency=2 THEN 3 WHEN 3~5 THEN 4 ELSE 5 END`。

### 7.2 修正后结果验证

| 分层 | 修正版 avg_monetary | 修正版占比 |
|---|---|---|
| 核心用户(R>=4,F>=4,M>=4) | 860 雷亚尔 | 0.006%(6 人) |
| 中坚用户 | 504 雷亚尔 | 0.2%(218 人) |

| 普通用户 | 202 雷亚尔 | 43.4% |

核心用户消费(860)高于流失用户(56) — 方向确认正确。

### 7.3 方法论总结:NTILE vs. 业务阈值

| | NTILE(5) | 业务阈值 |
|---|---|---|
| 适用场景 | 数据连续分布、均匀差异 | 数据强长尾、离散、大量同值 |
| 优势 | 自适应,不调参 | 业务含义清晰,可解释 |
| 劣势 | 同值被强行分桶(业务含义不稳定) | 需要业务理解来定阈值 |
| Olist 适用 | ❌ F 维度 96.9% 同值,NTILE 失效 | ✅ 按复购次数直接定档 |

## 七、面试 Q&A 沉淀

> 把这一轮发现压缩成 4 个可口播的故事,面试时拿来即用。

### Q1:你这个项目里做过数据质量检查吗?发现了什么?

A:做了 6 类检查(行数 / PK / FK / 时间 / 异常值 / 空值),发现 4 类问题:

1. **维度引用缺失** —— 13 条 product 引用 2 个未翻译类别 → 补维度
2. **录入缺失伪装真值** —— 4 条 weight_g = 0 → 改 NULL,避免污染统计
3. **时间逻辑反常** —— 23 条送达早于发货,分布显示主要是时区错位 → 视图排除
4. **数据 BUG 集群** —— 8 条 delivered 但无 customer_date,钻取发现 2 条是同一卖家同一时刻发货 → 物流回调批量丢失

收获:**数据异常不是均匀分布,是集群的**。找到一条异常要立刻追问"还有多少同源的"。

### Q2:你导入数据时遇到过坑吗?

A:遇到过 3 个经典坑:

1. **CSV 行尾错配**:Kaggle CSV 实际是 LF,VSCode 状态栏显示 CRLF 是误导(显示=保存目标格式,不是当前字节)。用 `xxd` 看 hex 才发现。解决:`LINES TERMINATED BY '\n'`
2. **LOAD DATA 多分号**:`LOAD DATA + SET` 是一条复合语句,中间加分号会让 LOAD DATA 提前结束,字段静默 NULL,但**行数对、不报错**。直到 NULL 比例异常才反推回来
3. **CSV 字段内换行符**:order_reviews 的评论字段有未规范化的 `\n`,LOAD DATA 解析时字段错位。改用 pandas + `to_sql`,因为 pandas 实现了完整的 CSV 状态机(RFC 4180),处理脏文本更鲁棒

收获:**LOAD DATA 性能极高但脆弱,脏数据走 Python**。这就是工业界 ETL 工具(Sqoop vs Spark)的设计哲学差异。

### Q3:你怎么判断哪些数据要修复、哪些要保留?

A:三类不同处理:

1. **维度缺失 → 补**(13 条类别翻译):事实是真实业务,维度是描述性属性,补维度不删事实
2. **录入伪值 → 规范化**(4 条 weight=0):0 不是合法重量,改 NULL 符合"未知 ≠ 假数据"原则
3. **业务异常 → 标记不删**(23 条时间反常):无法还原真实时间,改了等于伪造数据;创建清洗视图 `v_orders_clean`,**原始数据 + 清洗视图并存**,这是工业界数仓的标准做法

核心原则:**事实表不可改,维度表可补,异常用视图过滤**。

### Q4:你从 Olist 数据里看出什么业务规律?

A:订单 NULL 分布做交叉分析,得到一张订单生命周期地图:

- 三层漏斗:**未付款 0.16% → 未发货 1.79% → 未送达 2.98%**,与电商行业典型转化率匹配
- 关键发现:**609 单 unavailable(已付款但商品缺货)** —— 几乎 100% 导致退款 + 差评,这是卖家供应链管理的核心 KPI,可用作搜推算法的卖家筛选指标
- 评分 **U 型分布**(1 星 12% + 5 星 58%),反映评价幸存者偏差 —— 不能直接作回归 label,需做去偏处理

收获:**数据分析的价值不在跑数,在解读**。同样的 NULL 分布,初学者看到的是"哪些字段缺失",分析师看到的是"业务漏斗 + 卖家健康度"。

## 第八章:Cohort 留存 + LTV 分析记录(2026-06-04)

> 分析文件:`sql/35_analysis_cohort_ltv.sql`
> 状态:✅ 已完成

### 8.1 分析目的

弥补 RFM 的盲点——RFM 是"用户现在的状态",看不到时间维度上的变化趋势。Cohort 回答:同一批用户首购后第 N 个月,还有多少人回来?人均贡献多少 GMV?

### 8.2 方法论

**Cohort 定义**:按首购月份(`DATE_FORMAT(MIN(order_purchase_timestamp), '%Y-%m')`)分组。
**Period 计算**:`PERIOD_DIFF(订单月, 首购月)`,首购当月 period=0。
**LTV 口径**:已实现的历史人均累计 GMV,不是预测未来 LTV。

### 8.3 关键结果

| 指标 | 数值 | 解读 |
|---|---|---|
| 次月留存率(p1_retention) | 0.4%-0.7% | 每 200 新用户只有 1 人次月复购 |
| LTV 增幅 | 3-7 BRL(3%-5%) | 首单贡献用户价值的 96.4% |
| 首单 ARPU 稳定性 | 150-175 BRL | 各 cohort 用户质量一致 |
| 黑五 2017-11 | cohort +63%,ARPU -6% | 促销拉新有效但客单降低 |

### 8.4 与 RFM 的交叉验证

两个独立方法得出同一结论:

| 方法 | 指标 | 结论 |
|---|---|---|
| RFM | 96.9% 用户 frequency=1 | 绝大多数用户只买一次 |
| Cohort | 17 个月 LTV 只涨 3.7% | 复购几乎不贡献价值 |

**两个正交维度互相验证 → 结论置信度极高。**

### 8.5 方法踩坑

| 坑 | 问题 | 修正 |
|---|---|---|
| PERIOD_DIFF 格式 | 传 '2017-03' 带横杠参数错误 | 使用 `REPLACE(cohort_month, '-', '')` 转为 '201703' |
| **漏写 order_status 过滤** | Block 1 两处 JOIN 均未加 `WHERE o.order_status = 'delivered'`,导致 canceled/unavailable 等未完成订单被算进 cohort | 已在 cohort_base 和 cohort_orders 两处 CTE 中补上该条件,cohort_size 修正后减少约 3-14%(例:2016-10 从 301→258 人) |
| 窗口效应 | 早期 cohort final_ltv 更高 | 并非用户质量下降,而是观测窗口更长;应按同一 period 横向对比 |
| 数据截断 | 2018-08 cohort p1_retention=0.02% | 数据集截止 2018-08-29,次月窗口不完整,排除该点 |
| 口径一致性 | RFM 用 customer_unique_id + delivered | Cohort 必须严格一致,否则两张图对不上 |

### 8.6 面试金句

见 `面试回答.md` Part 7(5 条 Q&A):
- RFM vs Cohort:状态 vs 过程
- 0.5% 留存正常吗:获取型业务,低留存是商业模式而非问题
- 跨 cohort 对比:不能比 final_ltv,要比同一 period
- LTV 只涨 3-5%:量化了复购上限,说明拉新 ROI > 留存 ROI
- 常见坑:口径一致、窗口效应、数据截断、PERIOD_DIFF 格式

## 第十章:卖家健康度分析记录(2026-06-04)

> 分析文件:`sql/50_analysis_seller.sql`
> 状态:✅ 已完成（Block 1-3 + 集中度 + Bottom 10）

### 10.1 分析目的

构建卖家健康分体系，量化"谁是优质供给、谁拖累体验"，支撑流量分配和卖家治理决策。

### 10.2 方法论

**健康分公式**：
```
health_score = PERCENT_RANK(GMV) × 100
             + PERCENT_RANK(评分) × 100
             + PERCENT_RANK(备货天数 DESC) × 100
             - 缺货惩罚(> 5% → -20)
             - 取消惩罚(> 10% → -20)
             - 差评惩罚(> 25% → -20)
```

**CASE WHEN 口径控制**：每个指标在聚合函数内部独立控制口径——经营指标排除 canceled/unavailable，体验指标仅 delivered + 时间戳完整，风险指标用所有订单。

### 10.3 关键结果

| 指标 | 数值 |
|---|---|
| 活跃卖家 | 1,783（valid_orders >= 5） |
| Top 100 GMV 占比 | 46.44% |
| 5K+ GMV 卖家占比 | 86.09% |
| S 级卖家 | ~100 人（health_score >= 180） |
| D 级卖家 | ~300 人（health_score < 0） |

### 10.4 核心发现

**S 级卖家画像**：评分 4.6+、差评率 < 4%、备货 < 1.1 天、SP 州主导。

**最大卖家 ≠ 最好卖家**：GMV 第 1（1,159 单/16.3 万）健康分只排第 93。GMV 权重可能掩盖体验问题。

**office_furniture 烂卖家未进 Bottom 10**：GMV 分位过高抵消差评惩罚，暴露健康分权重的设计偏差。

**D 级全是小卖家**（5-16 单、评分 2.0-3.44），平台真正风险在 A/B 级中等卖家。

### 10.5 踩坑

| 坑 | 修正 |
|---|---|
| CASE WHEN 口径混乱 | 每个聚合函数内独立写 CASE WHEN，不再用 WHERE 整表过滤 |
| PERCENT_RANK 方向 | 备货天数加 DESC |
| NULLIF 缺失 | 差评率/缺货率/取消率三分母全加 NULLIF |
| 小卖家噪声 | `WHERE valid_orders >= 5` |
| GMV 权重过高 | 已标注为设计偏差，面试中主动讨论 |

### 10.6 面试金句

见 `面试回答.md` Part 9（5 条 Q&A）:
- 健康分构建：PERCENT_RANK 而非 NTILE，权重是业务选择
- 最大卖家 ≠ 最好卖家：GMV 排名 vs 健康分排名对比
- office_furniture 烂卖家未进 Bottom 10：暴露 GMV 权重过高
- 卖家集中度温和：供给多样性对推荐系统的影响
- D 级全是小卖家：健康分核心用户在 A/B 级

## 第十一章:策略收益估算记录(2026-06-04)

> 分析文件:`sql/60_strategy_impact.sql`
> 状态:✅ 已完成（4 个 Block）

### 11.1 分析目的

把品类分析和卖家分析的发现量化为 GMV 损失 + 治理 ROI，将分析结论转成业务优先级。

### 11.2 策略清单

| 策略 | 目标 | 指标 | 预期收益 | 风险 |
|---|---|---|---|---|
| 缺货/取消治理 | 挽回流失订单 | 挽回 GMV | ~20 万 BRL（1.3%） | unavailable 需系统投入 |
| 风险卖家限流 | 降低缺货率 | 缺货率降 20% | 仅 108 BRL | 只命中 1 个卖家 |
| 备货优化 0.5 天 | 减少等待时间 | 慢订单(>3天) | SP 州 8,597 天 | 用户感知不明显 |
| 早评机制调整 | 消除评分偏差 | 全平台评分 | +0.13 分 | 评论量短期下降 |

### 11.3 核心发现

**GMV 损失天花板 ~20 万 BRL（1.3%）**：
- canceled: 461 单/105,886 BRL（有数据，精确）
- unavailable: 6 单/2,140 BRL（有 items 数据）+ 估算 97,000 BRL（609 单 × 160 BRL）
- 609 单 unavailable 中仅 6 单有 order_items，损失被系统性低估

**缺货卖家归因重大修正（2026-06-04）**：
- 原版用 INNER JOIN order_items → 只找到 7 单 unavailable
- 修正后用 LEFT JOIN → 保留全部 609 单，但发现 **603 单无法归因到卖家**（无 order_items → 无 seller_id）
- 775 个无 seller 归属的订单中：603 unavailable + 164 canceled + 8 其他
- 排除 NULL seller 后：仅 1 个卖家触发 > 5% 门槛，且只有 1 单缺货
- **结论反转**：缺货治理在 Olist 数据中无法精确到卖家级别，这是数据固有限制，不是策略方向错误

**早评偏差 0.13 分**：8.63% 评价是早评（2.79 vs 4.28），消除后评分 4.16 → 4.29。

**策略优先级**：早评调整（P0/低投入高确定性）> canceled 治理（P1）> 备货预期管理（P2）> unavailable 治理（P3/高投入）。

### 11.4 踩坑/数据质量限制

| 问题 | 说明 | 面试处理 |
|---|---|---|
| unavailable 只有 6 单有 items | 缺货订单无商品行，无法精确算 GMV | 用 609 × 客单价均值估算，标注假设 |
| 仅 1 个卖家触发缺货门槛 | 不是方法错，是缺货分布长尾 | 说"结果推翻了假设"而非掩盖 |
| 评分提升无法直接转 GMV | 评分 → CVR 需 AB 实验验证 | 给 0.13 分提升，但不编造 GMV 换算 |

### 11.5 面试金句

见 `面试回答.md` Part 10（5 条 Q&A）:
- GMV 损失估算方法 + 数据质量限制的主动说明
- 缺货是长尾问题：分析推翻了假设
- 备货优化：预期管理比单纯加速重要
- 早评 0.13 分：区分偏差修正和真提升
- 四策略优先级排序：P0/P1/P2/P3 + 依据

## 第十二章:推荐系统候选召回原型 — 数据集构建(2026-06-04)

> 分析文件:`sql/70_reco_dataset.sql`
> 状态:✅ SQL 部分已完成（7 个 Block），Python baseline 待开发

### 12.1 分析目的

把数据分析阶段的"用户分层、品类体验、卖家健康度"整合为推荐系统的训练样本和评估框架。打通"数据分析 → 算法工程"的最后一环。

### 12.2 数据结构

| Block | 表名 | 内容 | 行数（预估） |
|---|---|---|---|
| 1 | `interaction_events` | 购买事件明细（事件级，不聚合） | ~11 万 |
| 2 | `train_interactions` | 训练集 user-item 聚合（< 2018-06-01） | ~9 万 |

| 7 | 品类热门 baseline | 每个品类 Top 10 | — |

### 12.3 关键设计决策与踩坑

**致命坑 1：先聚合再切分**
- 第一版用 `user_id + product_id` 聚合后用 `last_purchase_time` 切分。
- 5 月买 A、7 月再买 A → 整条 pair 因 `last_purchase_time >= 2018-06-01` 进入验证集，训练集无历史。
- **修正**：先事件级切分（`WHERE order_purchase_timestamp < '2018-06-01'`），再分别聚合 train 和 val。

**致命坑 2：商品特征用全量数据**
- 第一版 product_features 用了全量订单的 avg_price、avg_review_score、order_count。
- 热门召回和特征提前知道了"未来哪个商品会火"→ Recall@K 偏乐观。
- **修正**：Block 4 只用训练期数据（`WHERE order_purchase_timestamp < '2018-06-01'`）。

**坑 3：order_reviews 一单多评放大 order_items**
- 直接 LEFT JOIN order_reviews → 一单多评时 order_items 行被放大 → avg_price 等指标被拉伸。
- **修正**：先将 review 按 order_id 聚合（`AVG(review_score)`），再 JOIN。

**坑 4：评分是订单级不是商品级**
- Olist 没有商品级评分。用订单评分近似商品体验评分。已在注释中主动标注。

**坑 5：热门召回是全局 Top 200 而非品类 Top K**
- 第一版 `LIMIT 200` 不保证每个品类都有候选。
- **修正**：Block 6 全局热门 + Block 7 品类 Top 10（`ROW_NUMBER() OVER(PARTITION BY category_name)`）。

**坑 6：CREATE TABLE AS SELECT 不建索引**
- 后续 JOIN、ItemCF 计算会非常慢。
- **修正**：每个 CREATE TABLE 后加 PRIMARY KEY + 查询索引。

## 第十三章:Python 召回 Baseline 评估记录(2026-06-07)

> 文件:`scripts/reco_baseline.py`
> 状态:✅ 已完成

### 13.1 评估设计

双口径评估：
- **all_targets**：全量验证集（模拟真实线上，含用户和商品冷启动）
- **warm_targets**：只保留训练期已出现的商品（排除商品冷启动噪声）

空召回计入 0 分：CategoryHot 和 ItemCF 对冷启动用户召回为空时，不跳过，如实反映"召不回"。

### 13.2 评估结果

| 召回路 | all_targets Recall@50 | warm_targets Recall@50 | 冷启动用户 Recall@50 |
|---|---|---|---|
| GlobalHot | **3.73%** | **6.49%** | 3.75% |
| CategoryHot | 0.04% | 0.08% | **0** |
| ItemCF | **0** | **0** | **0** |

### 13.3 核心发现

**ItemCF 完全失效**：
- 商品共现对 11,940 → min_cooccur=2 过滤后 → 422 对（过滤 96.5%）
- 每个商品平均 1.3 个相似邻居
- 根因：96.9% 用户只买一次 → item-item 共现矩阵接近空集

**GlobalHot 是最强 baseline**：
- 不需要任何用户历史 → 对冷启动用户 Recall 不衰减
- warm_targets Recall@50 = 6.49%（模型能力的真实体现）

**CategoryHot 天然瓶颈**：
- avg_recall_count@50 仅 10.3 → 用户品类历史有限
- 对冷启动用户 = 0（无历史品类）

**Candidate Coverage 极低**：
- 三条召回路候选池合计 < 2%：GlobalHot 200 个、CategoryHot 472 个、ItemCF 318 个
- 实际 Recommendation Coverage@50 = 52 个商品（0.16%）

### 13.4 核心结论

在 Olist 上，冷启动策略（热门 + 品类 + 评分/履约特征过滤）比协同过滤更实用。这不是模型调参的问题，是数据模式决定的。97.72% 用户冷启动 + 96.9% 一次购买 = 协同过滤没有信号可捕捉。处理冷启动的能力在任何公司都是通用的。

- ItemCF Recall=0 不是 bug，是整个项目最重要的结论

## 第十四章:电商指标体系设计(2026-06-07)

> 文件:`notes/metrics_framework.md`  
> 状态:✅ 已完成

### 14.1 分析目的

把前面已经完成的大盘、漏斗、RFM、Cohort、品类、卖家、策略收益和推荐召回结果，统一收束为一套纯数据分析岗可讲的电商指标体系。

这个模块回答的不是"某个 SQL 怎么写"，而是数据分析岗最常见的问题:

```text
如果让你负责一个电商平台的数据分析，你会怎么搭指标体系?
```

### 14.2 指标体系结构

三层结构:

```text
北极星指标
-> 一级健康指标
-> 二级公式拆解
-> 三级诊断维度
```

北极星指标定义为:

```text
有效 GMV = delivered 订单的商品价格 + 运费
```

辅助北极星:

```text
有效成交订单数 = delivered 且存在 order_items 的订单数
```

这样设计的原因:

- GMV 代表平台交易规模，是最直接的商业结果。
- 有效订单数避免 GMV 被少数高客单价订单拉动。
- `canceled` / `unavailable` / 未送达订单不应计入真实成交。
- 与前面 RFM、Cohort、品类、卖家、推荐样本的 delivered 口径保持一致。

### 14.3 一级指标

| 一级指标 | 核心问题 | 代表指标 |
|---|---|---|
| 交易规模 | 平台卖得动吗 | GMV、订单量、买家数、AOV |
| 用户价值 | 用户会回来吗 | 复购率、Cohort 留存、LTV、RFM |
| 转化漏斗 | 订单有没有走完 | 付款率、发货率、送达率、取消率、缺货率 |
| 履约体验 | 等得久不久 | 付款耗时、备货耗时、配送耗时 |
| 评价体验 | 用户满意吗 | 平均评分、差评率、早评率、评分偏差 |
| 供给质量 | 卖家和品类健康吗 | 品类 GMV、卖家健康分、S 级卖家占比 |
| 增长效率 | 拉新是否有效 | 新用户数、首单 ARPU、Cohort 首购 GMV |

### 14.4 二级公式拆解

核心公式:

```text
GMV = 订单量 × 客单价
    = 活跃购买用户数 × 人均订单数 × 客单价
    = 活跃购买用户数 × 人均订单数 × (商品均价 + 运费)
```

履约公式:

```text
总履约时长 = 付款耗时 + 备货耗时 + 配送耗时
```

用户价值公式:

```text
用户 LTV = 首单 GMV + 复购 GMV
复购率 = 下单次数 >= 2 的真实用户数 / 真实用户总数
```

卖家健康公式:

```text
卖家健康分 = GMV 分位 + 评分分位 + 备货时效分位 - 风险惩罚
风险惩罚 = 缺货惩罚 + 取消惩罚 + 差评惩罚
```

### 14.5 三级下钻路径

| 下钻维度 | 回答问题 | 项目证据 |
|---|---|---|
| 时间 | 长期趋势还是短期波动 | 黑五 GMV 峰值、狂欢节发货波动 |
| 地域 | 哪些州贡献变化 | 早评率与评分的州级差异 |
| 品类 | 哪些品类拉动或拖累 | bed_bath_table、office_furniture |
| 卖家 | 是平台问题还是卖家问题 | office_furniture 70% 订单集中于单一卖家 |
| 用户 | 新老用户结构是否变化 | RFM、Cohort、首单品类 |
| 履约 | 是否由物流或备货拖累 | 平均履约 11.6 天 |
| 评价 | 是真实体验还是测量偏差 | 早评 8.63%，评分偏差 +0.13 |

### 14.6 常见坑

| 坑 | 说明 | 修正 |
|---|---|---|
| 把指标体系写成指标列表 | 只有 GMV/订单/评分，没有层级和归因 | 建立北极星 -> 一级 -> 二级 -> 三级结构 |
| GMV 口径混用 | 全量订单和 delivered 订单混算 | 明确有效 GMV = delivered + order_items |
| 忘记 `v_orders_clean` 不过滤 status | canceled/shipped 混入履约指标 | 必须额外 `WHERE order_status = 'delivered'` |
| 一单多商品放大订单数 | JOIN 后 count 行导致订单虚高 | 使用 `COUNT(DISTINCT order_id)` |
| 一单多评放大商品行 | review 直接 JOIN item 明细 | review 先按 order_id 聚合 |
| 只看 GMV 不看护栏 | GMV 上升但差评/取消也上升 | 配置履约、评价、供给护栏指标 |

### 14.7 面试金句

```text
指标体系不是指标列表，而是一套从结果指标到过程指标、从公式拆解到维度下钻的业务诊断地图。
在 Olist 里，我用 GMV 作为北极星，但不会只看 GMV；我同时设置履约、评价、取消、缺货、卖家健康分作为护栏指标，避免平台为了短期 GMV 牺牲长期体验。
```

## 第十五章:GMV 波动归因分析 SQL(2026-06-07)

> 文件:`sql/15_analysis_gmv_attribution.sql`  
> 结果:`notes/15_analysis_gmv_attribution_results.txt`  
> 状态:✅ SQL 已执行，结果已回填

### 15.1 分析目的

回答纯 DA 岗高频问题:

```text
GMV 为什么涨/为什么跌?
```

相比 `sql/10_analysis_overview.sql` 的月趋势，本模块进一步拆解 GMV 波动来源。

### 15.2 方法论

核心公式:

```text
GMV = 订单量 × 客单价
    = 活跃购买用户数 × 人均订单数 × 客单价
```

环比精确拆解:

```text
ΔGMV = 订单量效应 + 客单价效应 + 交叉项

订单量效应 = (本月订单量 - 上月订单量) × 上月 AOV
客单价效应 = (本月 AOV - 上月 AOV) × 上月订单量
交叉项     = 订单量变化 × AOV 变化
```

### 15.3 SQL 结构

| Block | 内容 |
|---|---|
| 1 | 月度经营基础指标:GMV、订单量、买家数、AOV、人均订单数 |
| 2 | GMV 环比波动归因:订单量效应、客单价效应、交叉项 |
| 3 | 订单量背后的用户结构拆解:买家数 vs 人均订单数 |
| 4 | 2017-11 黑五峰值归因 |
| 5 | 黑五品类贡献拆解 |
| 6 | 月份完整性检查,标注数据截断风险 |

### 15.4 实际结果与发现

**2017-11 黑五峰值是典型订单量驱动。**

| 月份 | GMV | 订单量 | AOV | GMV 环比 | 订单量环比 | AOV 环比 |
|---|---:|---:|---:|---:|---:|---:|
| 2017-10 | 751,117 | 4,478 | 167.73 | 7.14% | 7.90% | -0.71% |
| 2017-11 | 1,153,364 | 7,289 | 158.23 | 53.55% | 62.77% | -5.66% |
| 2017-12 | 843,078 | 5,513 | 152.93 | -26.90% | -24.37% | -3.35% |

黑五月 GMV 环比增长 **402,247 BRL**，其中订单量效应约 **471,503 BRL**，AOV 效应约 **-42,547 BRL**，交叉项约 **-26,708 BRL**。结论是:黑五增长来自“量”，不是来自“价”。

**订单量增长本质上由购买用户数驱动。**  
2017-11 订单数从 4,478 增至 7,289，买家数从 4,417 增至 7,183，人均订单数仍约 1.015。这与前面 RFM/Cohort 的低复购结论一致:Olist 的增长主要靠拉新或促销带来的购买用户数增加，而不是老用户频次提升。

**黑五增长贡献品类集中。**

| 品类 | 2017-11 GMV 增量 | 贡献占比 |
|---|---:|---:|
| bed_bath_table | 49,754 | 11.27% |
| health_beauty | 41,216 | 9.34% |
| furniture_decor | 39,224 | 8.88% |
| toys | 33,656 | 7.62% |
| watches_gifts | 33,364 | 7.56% |

**趋势解读需排除不完整月份。**  
2016-09、2016-10、2016-12、2017-01、2018-09、2018-10 被标记为不完整月份；正式趋势分析应主要看 2017-02 至 2018-08。

### 15.5 踩坑预警

| 坑 | 修正 |
|---|---|
| 只看 GMV 趋势不做归因 | 拆成订单量和 AOV |
| 把不完整月份当成经营下滑 | Block 6 标注月份完整性 |
| `v_orders_clean` 忘记 status 过滤 | 所有交易指标显式加 `order_status='delivered'` |
| 一单多商品导致订单数虚高 | `COUNT(DISTINCT order_id)` |

### 16.3 实际结果与发现

**SP 是绝对核心市场。**

前 5 州 GMV 占比约 **73.19%**，其中 SP 单州贡献 **37.41%**。这说明地域经营和物流优化不能平均用力，应该先看 SP/RJ/MG/RS/PR。

**跨州订单显著慢于同州订单。**

| 路由 | 订单量 | GMV | 平均配送天数 | 平均总履约天数 | 延迟率 | AOV |
|---|---:|---:|---:|---:|---:|---:|
| 同州 | 34,806 | 4,667,223 | 4.73 | 7.91 | 5.98% | 134.09 |
| 跨州 | 61,884 | 10,746,441 | 11.69 | 15.03 | 9.00% | 173.65 |

跨州订单 AOV 更高，但配送时长是同州的 **2.47 倍**，延迟率也更高。这是典型“高价值但体验风险更高”的供需匹配问题。

**北部/东北部物流压力明显。**  
RR、AP、AM、AL、PA 的总履约天数分别为 **29.36、27.17、26.40、24.52、23.75 天**。其中 AL 延迟率 **23.93%**，差评率 **21.32%**，是物流治理优先州。

**高价值待治理市场明确。**  
RJ、RS、BA 被标为高价值待治理市场。它们有较高 GMV，但履约或评分存在压力，适合做物流 SLA、预计送达时间展示、跨州线路优化。

### 16.4 与既有分析的关系

本模块承接第 4.6.4 节的早评地域分析。此前已发现:

- 21 个州全部存在 early_score < normal_score。
- 早评率越高的州，收货前评分越低。
- 不是"偏远 = 分低"，而是"超出用户预期 = 分低"。

地域经营分析会把这个发现扩展到 GMV、履约和供需匹配层面。

## 第十九章:评价文本 VOC 分析 SQL(2026-06-07)

### 19.1 分析目的

用用户评论文本补足评分指标的解释能力，回答:

```text
用户为什么给差评?
是物流慢、没收到货、商品质量、退款客服，还是包装问题?
```

评分只能告诉我们"体验不好"，VOC 能进一步解释"哪里不好"。

### 19.2 方法论

Olist 评论主要是葡萄牙语。本模块使用 SQL 关键词规则做轻量分类:

| 关键词 | VOC 分类 |
|---|---|

| nao recebi / nao chegou | 未收到货 |
| defeito / quebrado / danificado / qualidade | 商品质量问题 |
| reembolso / estorno / troca / devolver | 退款退换问题 |
| atendimento / resposta / suporte | 客服响应问题 |
| embalagem / pacote | 包装问题 |

这不是完整 NLP 模型，但可解释、可复现，适合 SQL 项目和 DA 面试。

### 19.3 SQL 结构

| Block | 内容 |
|---|---|
| 1 | 评论文本覆盖率与评分分布 |
| 2 | 差评关键词规则分类 |
| 3 | VOC 原因 x 评分 |
| 4 | 品类 x VOC 原因 |
| 5 | 地域 x VOC 原因 |
| 6 | 履约时长与物流类差评关系 |

### 19.4 实际结果与发现

**低分用户更愿意写评论。**

| 评分 | 评论数 | 有文本评论 | 文本覆盖率 |
|---:|---:|---:|---:|

| 2 | 3,151 | 2,145 | 68.07% |
| 5 | 57,328 | 20,548 | 35.84% |

这说明差评文本更适合做问题诊断；但也要标注选择偏差，因为愿意留言的人不是全体用户。

**差评中物流与未收到货是主要可识别原因。**

| VOC 原因 | 评论数 | 占比 | 平均评分 |
|---|---:|---:|---:|
| 其他/未识别 | 5,013 | 46.03% | 1.22 |
| 物流配送问题 | 2,394 | 21.98% | 1.20 |
| 未收到货 | 1,847 | 16.96% | 1.11 |
| 商品质量问题 | 816 | 7.49% | 1.25 |

21 天以上订单的物流投诉占比达到 **33.12%**，几乎是 <7 天订单的 2.7 倍。这验证了前面履约时长、早评和评分之间的关系。

### 19.5 踩坑预警

| 坑 | 说明 | 修正 |
|---|---|---|
| 把关键词分类当完整 NLP | SQL LIKE 只能覆盖明显词 | 标注为方向性洞察 |
| 忽略葡萄牙语变体 | nao/não、重音字符会影响匹配 | 同时匹配常见变体 |
| 只看差评不看覆盖率 | 留言用户有选择偏差 | 先计算文本覆盖率 |
| 订单级评论归因到商品 | 一单多商品会混淆原因 | 结果用于品类方向判断，不做商品级因果 |

### 19.6 面试金句

```text
评分告诉我用户不满意，VOC 告诉我用户为什么不满意。
我没有把 SQL 关键词分类包装成 NLP 模型，而是明确它是可解释的规则分析，
用于快速定位差评主题，再决定是否值得上更复杂的文本模型。
```

## 第二十章:推荐混合召回 Hybrid Baseline(2026-06-07)

> 文件:`scripts/reco_baseline.py`, `notes/reco_notes.md`  
> 状态:✅ 已完成并已运行验证

### 20.1 分析目的

在 GlobalHot、CategoryHot、ItemCF 三条单路召回之后，进一步补一个更贴近真实业务的混合召回 baseline:

```text
冷启动用户: GlobalHot
非冷启动用户: CategoryHot 优先，不足用 GlobalHot 补足
```

这么做的原因是前一轮结果已经证明:

- Olist 验证期用户冷启动率高达 97.72%。
- GlobalHot 对冷启动最稳。
- CategoryHot 只在非冷启动用户上有价值。
- ItemCF 因共现矩阵稀疏，Recall@K 全为 0。

### 20.2 新增召回策略

| 召回路 | 逻辑 | 业务含义 |
|---|---|---|
| HybridRecall | 冷启动 GlobalHot；非冷启动 CategoryHot + GlobalHot 补足 | 当前最推荐写进项目亮点 |
| QualityHybrid | 冷启动高评分热门；非冷启动 CategoryHot + 高评分热门补足 | 体验优先版本，适合作为 AB 实验假设 |

### 20.3 核心结果

| 口径 | 指标 | GlobalHot | HybridRecall | 结论 |
|---|---:|---:|---:|---|
| all_targets | Recall@50 | 0.0373 | 0.0375 | Hybrid 略优于单纯热门 |
| all_targets | NDCG@50 | 0.0110 | 0.0112 | 排序位置也略有提升 |
| warm_targets | Recall@50 | 0.0649 | 0.0654 | 去掉商品冷启动噪声后更稳定 |
| 非冷启动 warm 用户 | Recall@50 | 0.0478 | 0.0657 | Hybrid 的价值主要来自 warm user |

QualityHybrid 的 all_targets Recall@50 为 **0.0120**，明显低于 GlobalHot。这说明“高评分/低差评”重排会牺牲购买命中，不能直接当作纯召回最优方案。

### 20.4 业务解释

HybridRecall 的优势不是因为算法复杂，而是因为它匹配了用户状态:

```text
没有历史 -> 不强行个性化，用热门兜底
有历史 -> 先用历史品类做轻个性化
候选不足 -> 用热门补齐，保证推荐列表完整
```

这比单一路线更像线上推荐策略，也更适合面试表达。

### 20.5 踩坑预警

| 坑 | 说明 | 修正 |
|---|---|---|
| 只看单路召回 | GlobalHot 和 CategoryHot 各有短板 | 按用户状态混合 |
| 高评分热门直接替代热门 | Recall 明显下降 | 作为体验护栏或实验组，不作为主召回 |
| 忽略空召回 | CategoryHot / ItemCF 指标虚高 | 空召回计入 0 分 |
| 只讲 ItemCF 失败 | 容易显得项目停在负结论 | 用 Hybrid 把负结论转成可落地方案 |

### 20.6 面试金句

```text
ItemCF 失败不是项目失败，而是告诉我当前数据的主要矛盾不是相似度计算，
而是冷启动和低复购。基于这个发现，我把召回策略改成 Hybrid:
冷启动用户用 GlobalHot 兜底，非冷启动用户先走历史品类，再用热门补足。
这比继续调 ItemCF 参数更符合 Olist 的业务数据形态。
```
