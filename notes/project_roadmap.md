# Olist 项目后续路线图与整体逻辑

> 目标:把当前项目从“SQL 数据分析项目”升级成“电商平台交易增长、供给治理与推荐召回原型项目”。
> 使用方式:每做一个分析模块，都按“为什么做 -> 怎么分析 -> 业务价值 -> 面试怎么讲”的顺序沉淀。

---

## 1. 项目整体定位

这个项目不要只包装成 Kaggle 数据分析。更强的定位是:

**基于 Olist 巴西电商数据集，构建一个从数据质量治理、交易漏斗诊断、用户价值分层、品类与卖家经营，到推荐系统候选召回的完整电商分析项目。**

它对应中大厂更关心的三类能力:

| 能力 | 项目中如何体现 |
|---|---|
| 数据可信 | 建表、导入、质量检查、异常修复、外键约束、清洗视图 |
| 业务判断 | GMV、漏斗、用户、品类、卖家、物流、评分偏差 |
| 策略落地 | 复购提升、缺货治理、卖家健康度、推荐候选池、AB 实验设计 |

核心思路是:  
**先证明数据可信，再发现业务问题，再定位损耗环节，再提出策略，再设计验证方式。**

建议 F 分数规则:

```sql
CASE
    WHEN frequency = 1 THEN 1
    WHEN frequency = 2 THEN 3
    WHEN frequency BETWEEN 3 AND 5 THEN 4
    ELSE 5
END AS f_score
```

为什么这么做:

- Olist 用户复购率低，大多数用户 frequency 都是 1。
- 如果用 `NTILE(5)`，会把大量同频用户强行切成 5 组，业务含义不稳定。
- 业务阈值更适合这种强长尾、低复购的数据。

**Python 评估核心发现**：
- **ItemCF 完全失效**：共现对 11,940 → 过滤后仅 422 对 → Recall@K 全为 0。不是模型有问题，是 96.9% 用户只买一次导致共现矩阵接近空集。
- **GlobalHot 是最强 baseline**：Recall@50 = 3.73%（all）/ 6.49%（warm）。对冷启动用户（97.72%）Recall 不衰减，因为不依赖历史。
- **CategoryHot 只在非冷启动用户上有微弱优势**：warm_targets 非冷启动 Recall@20 = 3.39%，但对冷启动用户 = 0（无历史品类）。
- **HybridRecall 是当前最适合写进项目亮点的方案**：all_targets Recall@50 = 3.75%，warm_targets Recall@50 = 6.54%，均略高于 GlobalHot；在非冷启动 warm 用户上 Recall@50 从 GlobalHot 的 4.78% 提升到 6.57%。
- **QualityHybrid 体现业务取舍**：all_targets Recall@50 = 1.20%，低于 GlobalHot，说明高评分重排会牺牲购买命中，更适合作为首单体验 AB 实验假设，而不是纯召回最优方案。
- **候选池覆盖仍然很低**：HybridRecall 候选池 512 个商品，覆盖率 1.59%；这说明当前原型是“低覆盖高确定性”的召回 baseline，后续若要工业化，需要扩展内容特征、卖家健康、价格段和履约特征。

输出重点:

- 明确说明 Olist 缺少曝光和点击数据，所以这里只做购买行为召回原型。
- 面试优先讲 HybridRecall，而不是继续强调 ItemCF；前者代表策略闭环，后者代表数据边界识别。
- 把品类、卖家健康度、物流、评分作为后续排序特征候选。

## 6. 最终面试故事线

可以用下面这条线讲完整项目:

```text
我没有直接做 EDA，而是先从数据可信开始。

第四步，我从供给侧做品类和卖家分析，识别哪些品类值得推荐、哪些卖家拖累体验，以及 unavailable 订单造成的 GMV 损失。

```

**我能把数据质量、业务诊断、策略设计和算法原型连成一个完整的电商平台问题解决过程。**
