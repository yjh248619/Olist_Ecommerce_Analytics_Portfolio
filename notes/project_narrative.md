# Olist 项目面试叙事稿

> 用途: 15-20 分钟项目讲解主线  
> 版本: 2026-06-07  
> 适用岗位: 数据分析岗 / 电商 DA / 搜推算法候选项目补充

## 一句话定位

这个项目不是普通 Kaggle EDA，而是一个模拟电商平台真实工作流的完整分析项目:

```text
数据可信
-> 交易诊断
-> 用户生命周期
-> 供给治理
-> 策略收益
-> 推荐召回原型
-> 指标体系 / 看板 / AB 实验
```

我想证明的不只是会写 SQL，而是能把数据质量、业务诊断、策略设计和验证闭环串起来。

## 1. 从数据可信开始

我没有一上来直接画图，而是先判断数据能不能信。

我用 MySQL 设计了 9 张原始表，使用 `LOAD DATA INFILE` 导入数据，同时补了主键、外键和索引。这个过程中遇到几个真实工程问题:

- `LOAD DATA` 出现过“行数正确但时间字段全 NULL”的假成功，根因是 `SET` 子句前多了分号。
- `order_reviews` 评论文本里有换行，直接用 SQL 导入会字段错位，所以改用 Python pandas 导入。
- 商品品类翻译缺失不是删订单，而是补维度表。
- 23 条送达早于发货的异常订单不直接改原表，而是创建 `v_orders_clean` 清洗视图。

面试讲法:

```text
我把原始事实表保留下来，用清洗视图承载分析口径。
这样既保证可追溯，也避免为了分析方便破坏原始数据。
```

## 2. 交易大盘与漏斗

在可信数据基础上，我先看平台大盘。

核心结果:

- 总 GMV 约 **15,843,553 BRL**。
- 有效订单约 **98,666 单**。
- AOV 约 **160.58 BRL**。
- 平均总履约约 **11.6 天**，其中备货约 **2.32 天**，配送约 **8.88 天**。

这里的业务判断是:物流耗时最长，但备货更可控。平台短期更容易推动卖家备货时效治理，而不是直接重建巴西物流网络。

## 3. GMV 波动归因

为了更贴近 DA 岗，我补了 GMV 归因框架:

```text
GMV = 订单量 × AOV
    = 购买用户数 × 人均订单数 × AOV
```

最典型的案例是 2017-11 黑五:

| 指标 | 2017-10 | 2017-11 | 变化 |
|---|---:|---:|---:|
| GMV | 751,117 | 1,153,364 | +53.55% |
| 订单量 | 4,478 | 7,289 | +62.77% |
| AOV | 167.73 | 158.23 | -5.66% |

结论:黑五 GMV 峰值不是客单价涨出来的，而是订单量拉动；订单量背后主要是购买用户数增长，人均订单数仍接近 1。

面试讲法:

```text
这是典型促销拉新驱动增长。
GMV 涨了不一定健康，要拆出量和价，判断是不是靠降价换来的。
```

## 4. 用户生命周期

用户侧我做了 RFM 和 Cohort 两套独立验证。

核心结果:

- 真实用户约 **96,096**。
- 复购率约 **3.1%**，96.9% 用户只买一次。
- 次月留存率约 **0.4%-0.7%**。
- 17 个月 LTV 增幅仅 **3%-5%**，首单贡献绝大部分用户价值。

这里我没有机械套 `NTILE`。因为 Olist 低复购非常明显，大量用户 frequency = 1，强行分 5 档会制造伪差异。所以 F 维度改用业务阈值。

面试讲法:

```text
RFM 不是公式套用，而是要适配数据分布。
当 96% 以上用户只买一次时，用户增长的重点不是复杂复购运营，而是首单体验和冷启动承接。
```

## 5. 品类与卖家供给治理

供给侧我先看品类，再钻到卖家。

品类层面:

- Top 5 品类贡献 **39.24% GMV**。
- Top 10 品类贡献 **62.19% GMV**。
- `bed_bath_table` 是大流量入口，但评分和差评率有压力。
- `office_furniture` 表面看是低评分、慢履约品类。

关键转折在 `office_furniture`:

继续钻到卖家后发现，问题不是品类天然差，而是 **70% 订单集中在一个低评分、慢履约卖家手里**。同品类里也存在评分更高、履约更快的卖家。

面试讲法:

```text
分析不能停在“哪个品类差”，因为品类不是可执行责任对象。
我继续下钻到卖家，才找到了真正能治理的对象。
```

## 6. 策略收益估算

我把分析结论进一步转成业务收益估算。

核心结果:

- canceled 损失约 **105,886 BRL**。
- unavailable 估算损失约 **97,000 BRL**。
- 两者合计 GMV 损失天花板约 **20 万 BRL**，约占平台 GMV **1.3%**。
- 早评偏差消除后，平台评分理论上可从 4.16 提升到 4.29，约 **+0.13 分**。

但我也主动标注限制:609 单 unavailable 中 603 单没有 `order_items`，无法精确归因到商品或卖家。

面试讲法:

```text
有些分析结论不是“马上治理谁”，而是“当前数据链路还不支持这个治理动作”。
主动指出数据边界，比硬编一个不可靠结论更专业。
```

## 7. 地域经营与 VOC

地域分析发现:

- SP 单州贡献 **37.41% GMV**，是核心健康市场。
- RJ、RS、BA 是高价值待治理市场。
- 跨州订单平均配送 **11.69 天**，同州订单 **4.73 天**。
- 跨州订单 AOV 更高，但延迟率也更高。

VOC 文本分析进一步解释了低评分原因:

- 1 星评论文本覆盖率 **76.55%**，5 星只有 **35.84%**。
- 差评中物流配送问题占 **21.98%**，未收到货占 **16.96%**。
- 21 天以上订单物流投诉占比 **33.12%**，约为 <7 天订单的 2.7 倍。

面试讲法:

```text
评分告诉我用户不满意，VOC 告诉我为什么不满意。
地域、履约和 VOC 交叉后，可以看到低分很大一部分来自物流体验，而不只是商品质量。
```

## 8. 推荐召回原型

最后我把项目连接到推荐系统。

我先明确数据边界:Olist 没有曝光、点击、搜索词、加购和实验分组，所以我不把它包装成完整工业推荐系统，而是做基于购买行为的离线候选召回原型。

我实现了:

- GlobalHot
- CategoryHot
- ItemCF
- HybridRecall
- QualityHybrid

核心结果:

- 用户冷启动率 **97.72%**。
- 商品冷启动率 **60.48%**。
- ItemCF 共现对 11,940，过滤后仅 422 对，Recall@K 为 0。
- GlobalHot 是最强单路 baseline，warm_targets Recall@50 = **6.49%**。
- HybridRecall 进一步提升到 **6.54%**。
- 非冷启动 warm 用户上，HybridRecall Recall@50 = **6.57%**，高于 GlobalHot 的 **4.78%**。

面试讲法:

```text
ItemCF 失败不是项目失败，而是数据诊断结果。
它说明 Olist 的主要矛盾不是相似度公式，而是冷启动和低复购。
所以我把策略转成 HybridRecall:冷启动走 GlobalHot，非冷启动走 CategoryHot + GlobalHot 补足。
```

## 9. 指标体系、看板与 AB 实验

最后我把分析沉淀成 DA 岗常见产出:

- `notes/metrics_framework.md`: 北极星指标、一级/二级/三级诊断指标树。
- `notes/dashboard_design.md`: 经营、履约、品类、卖家四类看板。
- `notes/ab_test_design.md`: 评价邀请时机、卖家健康加权、缺货供给降权、冷启动推荐四个实验。

这里的核心观点是:

```text
分析不是终点，验证才是闭环。
策略上线前要看核心指标，也要看护栏指标。
```

## 10. 最终总结

这个项目最后可以收束成一句话:

```text
Olist 的核心问题不是单一指标差，而是低复购、高冷启动、长尾供给和跨州履约共同作用。
所以更合理的策略不是一上来做复杂模型，而是先做好首单体验、供给治理、冷启动召回和 AB 验证闭环。
```

如果面试官追问“你最满意的地方”，可以回答:

```text
我最满意的是这个项目没有停留在 EDA。
它从数据质量开始，到业务诊断、策略收益、推荐原型、指标体系和 AB 实验，形成了一个完整的问题解决链路。
```
